## 零、市场与行业背景：为什么需要向量检索硬件加速

### 0.1 向量数据库市场爆发

向量数据库市场正处于高速增长期，背后是多股技术浪潮的叠加：

| 指标 | 数据 | 来源 |
|------|------|------|
| 向量数据库市场（2025） | **26.5 亿美元** | MarketsandMarkets |
| 向量数据库市场（2030） | **89.5 亿美元** | MarketsandMarkets |
| CAGR（2025–2030） | **~27.5%** | MarketsandMarkets |
| VDaaS 子市场（2025→2030） | 16.2 亿美元 → 61 亿美元，CAGR **30.2%** | 商业研究公司 |
| Agent 驱动向量数据库（2025→2031） | 4.6 亿美元 → 17.3 亿美元，CAGR **24.86%** | Mordor Intelligence |
| 面向 GenAI 的向量数据库（2025→2030） | 7.6 亿美元 → 19.1 亿美元 | GII Research |

**核心驱动力**：

- **RAG 和 LLM 落地**：企业需要将私有知识注入大模型，向量检索是 RAG 管线的核心环节
- **AI Agent 爆发**：Qdrant CEO Andre Zayarni 指出——"人类每几分钟做几次查询，Agent 每秒做数百甚至数千次查询"（2026），Agent 工作负载将检索变成了紧密内循环
- **非结构化数据增长**：企业 80% 以上数据为非结构化（PDF、邮件、日志、图片、视频），传统搜索无法有效利用
- **从独立向量 DB 到定制检索栈**：VentureBeat 2026 Q1 调查显示，混合检索采纳意愿从 10.3% 跃升至 33.3%；独立向量数据库（Pinecone、Milvus、Weaviate、Qdrant）**采纳份额集体下降**，企业转向定制检索栈（2026 Q1 已达 35.6%）

### 0.2 规模膨胀——从 10M 到 10B，每跨越一个数量级架构就必须改变

市场增长（$26.5 亿→$89.5 亿）的直接结果是**客户的向量数据规模每 12-18 个月翻一番**。但向量检索的架构不是"线性伸缩"的——每跨越一个规模量级，原有的架构假设就会失效，必须引入新的硬件加速手段。

**向量规模的分层效应**：

| 规模层级 | 典型需求 | 可用的搜索架构 | 硬件角色 | 代表场景 |
|---------|---------|--------------|---------|---------|
| **≤10M**<br>几亿~几千万 | 单机可部署<br>~15 GB 内存 | HNSW（CPU）<br>IVF（CPU） | 纯 CPU 足够<br>无需专用硬件 | 初创公司 MVP<br>百/千级用户的 RAG |
| **10M~100M**<br>几千万~几亿 | 单机可能不够<br>~150 GB 内存 | IVF-PQ 压缩<br>DiskANN（SSD 卸载） | GPU 可加速构图<br>CPU 胜任查询 | 中型企业知识库<br>万级用户的 RAG |
| **100M~1B**<br>几亿~十亿 | 单节点显存不足<br>~1.5 TB 原始数据 | IVF-PQ/SQ 量化（必须）<br>多节点分布式图索引 | **GPU 构图成为刚需**<br>近存储（FPGA/SSD）开始有价值 | 企业级搜索<br>Agent 工作负载 |
| **1B~10B**<br>十亿~百亿 | 分布式必须<br>~15 TB 原始数据 | 分层索引 + 量化 + 分布式 | **近存储 / 存内计算成为差异化方案**<br>GPU 仅做构图，不参与在线搜索 | IBM CAS 100B<br>大型 RAG + Agent 系统 |
| **≥10B**<br>百亿+ | 专用的多级存储<br>>15 TB | 存储计算融合<br>精确/ANN 双轨 | FPGA（VSM 48× 扩展）<br>ASIC（Spyre 精确搜索） | 超大规模企业<br>全量企业知识搜索 |

**关键洞察：每个规模层级需要的硬件不同**

| 过渡点 | 技术上发生了什么 | 对应硬件需求 |
|--------|---------------|------------|
| **10M→100M** | 单 GPU VRAM（H100 80GB）已无法容纳全部原始向量 | PQ/SQ 量化进入部署必须项；GPU 构图加速（CAGRA 45s vs 22min）开始体现价值 |
| **100M→1B** | CPU HNSW 构图从小时级变为天级，不可接受 | GPU 构图不再是"更好"而是"必须"；FPGA 近存储方案（VSM/SmartANNS）展示差异化 |
| **1B→10B** | 纯 GPU 搜索因 VRAM 限制被迫退出在线路径 | GPU 退化为构图工具；FPGA（VSM 48× 扩展）或 NAND 内计算（NDSEARCH）进入主力 |
| **10B→100B+** | 任何单节点架构都不可行；能耗成为关键约束 | 存算融合系统（CAS）+ 精确搜索 ASIC（Spyre）+ 近存储 FPGA（VSM）多级协同 |

**这对硬件加速意味着什么**：

```
2022 年：大多数部署在 10M-100M 级别 → GPU 是"可选项"
2024 年：主流部署向 100M-1B 迁移 → GPU 构图成为刚需
2026 年：头部企业触及 1B-10B → FPGA/ASIC 近存方案开始证明价值
2028 年（预测）：百亿级普及 → 存算融合系统成为主流
```

> **核心论点**：向量数据库的市场增长不是线性推高了对 GPU 的需求——它**逐级突破架构上限**，让过去"够用"的 CPU 和 GPU 方案在更大规模下失效。每个新量级都引入了"必须用新硬件才能解决"的问题：10M→100M 引入了量化需求，100M→1B 引入了构图瓶颈，1B→10B 引入了数据搬运瓶颈。**硬件加速方案的出现时机不是巧合——它们是规模逼出来的**。

### 0.3 规模膨胀带来的技术瓶颈——每个量级催生新的痛点

上一节展示了规模膨胀的规律。现在逐一检视每个规模量级带来哪些具体的技术痛点——**注意每个痛点的"触发量级"，理解为什么小规模时不是问题，大规模时才成为瓶颈**。

| # | 痛点 | 触发量级 | 具体表现 | 关键数据 |
|---|------|---------|---------|---------|
| **① 数据规模爆炸** | **≥100M 向量起成为硬约束** | 单 GPU HBM 无法容纳十亿级向量 | 1B 个 768 维 FP16 向量 ≈ **3 TB**，远超 H100 80GB / B200 192GB |
| **② RAG 延迟反转** | **≥100M 向量起，搜索超过 LLM 推理** | 向量搜索成为首 Token 延迟（TTFT）主导 | 在 128M 个 2048 维向量上，搜索占 TTFT **85.8%**（VectorLiteRAG, 2025）；CPU 搜索比 LLM 预填充慢 **2×** |
| **③ 内存墙** | **所有规模均存在，但 ≥1B 时才致命** | 向量搜索是纯内存密集型，而非计算密集型 | 算术强度仅 **0.5–0.75 FLOPs/byte**，完全受限于内存带宽（HBM3 约 1.5 TB/s vs PCIe 5.0 32-64 GB/s） |
| **④ GPU 资源竞争** | **≥100M 向量起，索引与 LLM 争显存** | 向量索引与 LLM KV Cache 争抢显存 | 加载 17GB 向量索引使 KV Cache 大幅缩减 → LLM 生成吞吐显著下降 |
| **⑤ 索引构建慢** | **≥1B 向量起，CPU 构图以天计不可接受** | CPU 上亿级构图需数小时到数天 | CAGRA GPU **45 秒** vs CPU HNSW **18-22 分钟**（10M 1024D）；IBM CAS 100B 索引 CPU 需 **120 天** |
| **⑥ TCO 与能耗** | **≥100M 向量起，GPU 成本优势开始反转** | GPU 集群成本高昂，纯 CPU 方案性能不足 | 百度 1536D-5M 实测：48 CPU 实例 $28,800/月 vs 6 GPU 实例 **$3,600/月**（8 倍成本降，4.9× QPS 提升）|
| **⑦ 规模化的延迟抖动** | **≥1B 向量 + 写混合负载起** | P99 延迟在混合查询和写负载下飙升至数秒 | 实时 RAG / Agent 场景需要 **sub-10ms** 稳定搜索延迟 |
| **⑧ 数据搬运开销** | **≥1B 向量起，冯·诺依曼瓶颈主导** | 数据搬运取代计算成为主要矛盾 | 所有硬件方案的共同设计共识：**减少数据搬运比增加算力更关键** |

**从规模看痛点优先级**：

```
10M 级以下：痛点 ③（内存墙）是唯一存在的理论问题，实际 CPU 够用
    ↓
100M 级：①④⑥（数据规模、资源竞争、TCO）开始显现 —— 这是 GPU 方案首次进入决策
    ↓
1B 级：②⑤⑦⑧（RAG 延迟、构图慢、延迟抖动、数据搬运）集中爆发 —— GPU 开始不够用
    ↓
10B 级以上：所有 8 个痛点全部激活，GPU 退出在线路径，近存储/存算融合成为必须
```

> **硬件加速方案的出现时机不是巧合——它们是被规模逼出来的**。10M 级时 CPU HNSW 够用，不会有 VSM 的市场需求；100M 级时构图开始慢，CAGRA 有了价值；1B 级时数据搬运成为主导瓶颈，VSM/NDSEARCH/HERMES 的"近数据计算"才成为有意义的替代方案。

### 0.4 向量检索的计算特征——为什么堆算力不是答案

向量最近邻搜索（无论是精确暴力搜索还是 ANN 索引搜索）的**算术强度**（Arithmetic Intensity，FLOPs/byte）极低：

| 工作负载 | 算术强度 | 限制因素 |
|---------|---------|---------|
| 向量搜索（L2/IP） | **~0.5–0.75 FLOPs/byte** | 🔴 **内存带宽** |
| CNN 推理 | 10–50 FLOPs/byte | 🟢 计算 |
| LLM 推理（decode） | 5–20 FLOPs/byte | 🟡 混合 |
| LLM 推理（prefill） | 50–200+ FLOPs/byte | 🟢 计算 |

**这意味着**：

- 堆更多 FP32/TF32 算力（GPU Tensor Core 数千 TFLOPS）**对向量搜索几乎无帮助**
- 核心加速路径只有一条：**缩短数据搬运路径**，让计算发生在尽可能靠近数据的地方
- 这解释了为什么 **NDSEARCH（NAND 内计算）**、**IBM VSM（SSD 控制器内流水线）**、**DReX（CXL 近存计算）**、**IBM HERMES（模拟存内计算）** 都以"消除数据搬运"而非"增加算力"为设计目标
- 这也解释了为什么 **Falcon（FPGA）** 对比 V100 能实现 **26.9× 能效提升**——不是因为 FPGA 算力更强，而是因为数据路径更短

### 0.5 产业界共识与分歧

各主流向量数据库厂商正在以不同路径回应硬件加速挑战：

| 厂商 | 领导层观点 | 硬件策略 | 技术路线 |
|------|----------|---------|---------|
| **Pinecone** | 创始人 Edo Liberty（2024）直言"GPU 上正在发生的事情是不可持续的" | 无服务器架构，**存储与计算分离**，突发负载成本据称降低 10-100× | 软件优化优先 |
| **Qdrant** | CEO Andre Zayarni（2025-2026）指出 Agent 工作负载打破现有架构 | 基于 **Vulkan API** 的硬件无关 GPU 加速，避免 CUDA 锁定，支持 NVIDIA/AMD/Intel/Apple Silicon | GPU 通用 + 硬件无关 |
| **Milvus/Zilliz** | CEO Charles Xie（2025）强调"以存储级成本实现十亿级向量搜索" | 混合 CAGRA（GPU 索引构建 **12-15× 更快** + CPU 查询服务）+ 与 Pliops LightningAI 合作近存储加速 | 混合方案（GPU 构建 + CPU 推理 + 近存储） |
| **IBM** | 走最深度的硬件自定义路线 | CAS（H200+存储系统）、VSM（FPGA 近存）、Spyre（ASIC 暴力搜索）、HERMES（模拟 PCM 存内计算） | **全栈硬件自研** |
| **Samsung / SK Hynix** | 从存储制造端推动变革 | SmartSSD（FPGA 近存）、NDSEARCH（NAND 内 LUN 级 ASIC）、AiMX（GDDR6-PIM） | **存储内计算** |

**行业趋势信号**：

1. **硬件加速走向多样化**：不再仅依赖 NVIDIA GPU，Vulkan 支持 AMD/Intel/Apple，FPGA、ASIC、PIM、CXL 多路径并行
2. **GPU 并非万能解**：GPU 在索引构建上优势明确（45s vs 18-22min），但查询服务的 TCO 优势需结合具体吞吐量判断——GPU 硬件通常占自托管预算 **80%+**
3. **Agent 工作负载是新的压力测试**：Qdrant 指出 Agent 的三种故障模式——文档规模漏检、写负载段退化、分布式级联延迟
4. **运营可靠性超越功能**：2026 年 Q1 调查中，**31.1%** 企业将运营可靠性列为向量数据库头号关注点，超过访问控制

> **小结**：向量检索的硬件加速不是"要不要"的问题，而是"用哪条路径"的问题。数据规模增长指数级（CAGR 27.5% 的市场驱动），算力红利被内存带宽瓶颈和规模成本抵消——**FPGA 的近存计算、ASIC 的极致能效、GPU 的索引加速** 三大路线在不同维度上回应了同一个挑战：**如何在可控功耗和成本下，让向量搜索的延迟追上数据规模的增长**。

**深度洞察——如何审慎阅读本报告的市场数据**：本节引用的 MarketsandMarkets、Mordor Intelligence 等市场研究报告存在一个产业界共识但鲜少公开讨论的问题——这些报告的 CAGR 估算通常基于对供应商的访谈而非终端用户调研，存在系统性高估（"每个数据库都需要向量搜索"的厂商叙事可能会将替换性需求计入增量市场）。更可靠的信号来自终端行为：VentureBeat VB Pulse 的独立调查显示独立向量数据库的采纳份额在下降（2026 Q1 从峰值约 40% 降至约 35%），企业转向定制检索栈。这意味着**向量数据库的硬件加速需求并非均匀分布**——定制检索栈选择 CPU+NVMe+DiskANN 而非 GPU 加速的比例正在上升。这对硬件加速方案的市场预测有直接影响：如果向量搜索的主流部署形态从"GPU 加速的独立向量数据库"转向"CPU 查询的多模态存储系统"，那么 GPU 专用加速方案的 TAM 可能被结构性压缩，而 FPGA 近存储方案的机会窗口则可能延长。

## 向量检索算法阶段说明

以下为本报告使用的高维向量近似最近邻搜索（ANNS）算法的典型阶段划分，各加速器在不同阶段发力。

| 阶段 | 说明 | 计算特征 | 内存访问特征 |
|------|------|---------|------------|
| **① 索引构建** | 将数据集组织为图（HNSW/NSG）、倒排（IVF）、哈希或量化索引 | 计算密集型，需大量距离计算 | 顺序+随机混合 |
| **② 索引导航/图遍历** | 在索引结构中定位候选区域（如 HNSW 的逐层搜索） | 逻辑密集，距离计算+比较 | 随机访问主导 |
| **③ 候选点筛选** | 对候选集合进行粗过滤（如 IVF 的倒排链扫描） | 带宽密集型 | 顺序扫描 |
| **④ 距离计算** | 计算查询向量与候选向量的距离（L2/Cosine/IP） | **核心瓶颈**，向量化友好 | 带宽密集型 |
| **⑤ Top-K 排序** | 维护 K 个最近邻结果集 | 计算/逻辑混合，K 通常较小 | 寄存器/片上 SRAM |
| **⑥ 重排序/精确计算** | 对 ANN 候选集进行精确距离计算提升召回 | 计算密集型 | 取决于候选集大小 |
| **⑦ 向量量化/解码** | PQ/SQ 码本的解码与部分距离计算 | 查表+加法，计算密集 | 码本在片上 |

### 1.1 IBM CAS 内容感知存储

IBM 的内容感知存储（Content-Aware Storage）不是一个单独的加速芯片，而是 **将向量检索全流程嵌入企业存储系统** 的整体方案。GPU（NVIDIA H200）在其中承担 **索引构建 + 嵌入生成** 的加速角色。

#### 加速定位

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **① 索引构建** | ✅ **主加速点** | 6× H200 将索引构建时间从 **120 CPU天 降至 4 GPU天**（30× 加速）；未来集成 cuVS 目标 **1 天内完成 100B 索引** |
| **② 嵌入生成** | ✅ **主加速点** | GPU 运行 Granite Embedding 模型（通过 NVIDIA NIM 微服务）进行文档向量化 |
| **③ ANN 搜索** | ⚠️ 存储层卸载 | 搜索在存储层内完成（CPU + NVMe），GPU 仅加速索引，不参与在线查询 |
| **④ 精确暴力搜索** | ✅ IBM Spyre（见 ASIC 路线） | 100% recall 场景交由 Spyre ASIC 处理 |
| **⑤ 权限继承** | — | 向量 ACL 自动继承源文件 ACL（架构创新，非计算加速） |

#### 系统架构

```
文件写入 → IBM Storage Scale (GPFS)
   → Active File Management (AFM) 检测变更
   → Docling 解析文档（PDF/PPT/邮件 → JSON/Markdown）
   → Granite Embedding 模型（NVIDIA H200 GPU + NIM 加速）
   → 分层 ANN 索引（多独立子索引，错误隔离、增量更新）
   → 查询：GPU 嵌入查询向量 + 存储层 ANN 搜索 + LLM 上下文注入
```

#### 关键性能指标

| 指标 | 数据 | 加速对象 |
|------|------|---------|
| 向量规模 | **1000 亿**（384 维 FP32，153 TiB） | — |
| 查询延迟（端到端） | **694ms** @ 90% recall | 搜索全流程 |
| GPU 索引构建 | **4 天**（6× H200） | **① 索引构建** |
| CPU 索引构建基线 | ~120 天（双路 Intel） | ② 对比基线 |
| NVIDIA cuVS 集成目标 | 1 天内索引 **>100B** | ① 索引构建 |
| 每瓦性能（Spyre） | **8.8× vs H100** | ④ 距离计算/⑦ 嵌入推理 |

#### 关键硬件规格

- **存储设备**：IBM Storage Scale System 6000（4U，340 GB/s 读吞吐，175 GB/s 写吞吐）
- **GPU**：6× NVIDIA H200（GPUDirect Storage 直连 NVMe）
- **SSD**：48× Samsung PM9D3a PCIe Gen5（12 GB/s 顺序读/块）
- **网络**：400Gb InfiniBand + BlueField-3 DPU

> **来源**：[IBM Research Blog](https://research.ibm.com/blog/cas-100-billion-vector-storage-ai) | [IBM Research CAS](https://research.ibm.com/blog/content-aware-storage-generative-ai) | [SDxCentral](https://www.sdxcentral.com/news/ibm-and-nvidia-tout-content-aware-storage-on-100-billion-vector-database/)

**价值与代价**：
- ✅ **价值**：**唯一将向量搜索嵌入企业存储层而非加速计算层的方案**——继承了存储系统的可靠性、权限控制和数据治理；100B 向量（153 TiB）的演示达到 **694ms / 90% recall**，是公开已知最大规模向量搜索演示之一；GPU 仅用于构图（**30× 加速**：120 CPU 天 → 4 GPU 天），规避了 GPU VRAM 瓶颈
- ⚠️ **代价**：**694ms 端到端延迟**比 GPU（~5ms）慢 **~140×**、比 FPGA（<10ms）慢 **~70×**——企业级可靠性的代价是实时性的牺牲；硬件绑定 IBM Storage Scale System 6000 + FlashCore Module（单个 4U 节点 **~$100K+**），迁移成本极高；694ms 是整体端到端延迟（含嵌入生成和 LLM 推理），纯搜索部分可能在 **~20-50ms**，但未单独报告

### 1.2 NVIDIA CAGRA —— GPU 原生图索引的架构突破

**论文**：Ootomo et al., "CAGRA: Highly Parallel Graph Construction and Approximate Nearest Neighbor Search for GPUs," ICDE 2024 (arXiv:2308.15136)

CAGRA（CUDA Accelerated Graph-based Retrieval Algorithm）是从零为 GPU 并行架构设计的图索引算法，其设计理念与 CPU 时代的 HNSW 有本质差异。

#### CAGRA vs HNSW 的设计差异

| 设计选择 | CAGRA（GPU原生） | HNSW（CPU原生） |
|---------|---------------|----------------|
| **图结构** | 单层、固定出度 | 多层（层叠 NSW） |
| **边度** | 固定（`graph_degree`） | 可变（≤ max M）|
| **搜索起点** | 随机采样 N 个节点 + 距离排序 | 顶层入口点，逐层贪心下探 |
| **访存模式** | **规则对齐** → 适合 GPU 合并访问 | 随机 → 适合 CPU cache 预取 |
| **线程模型** | warp-level 协作（无分支发散） | 单线程/向量化 SIMD |

#### 构图（两阶段）

**阶段一：初始 k-NN 图**，三种策略自动选择：
- **NN-Descent**（高召回，高显存）：初始度设为最终度的 2-3×
- **IVF-PQ**（默认，GPU原生，更快，低内存）
- **AUTO**：根据数据集和 GPU 自动选择

**阶段二：图优化（剪枝）**，两项技术创新：

| 技术 | 说明 | 效果 |
|------|------|------|
| **Rank-based reordering** | 用邻居排序的 rank 替代距离值做剪枝，避免 `N×d_init×(d_init-1)` 次距离计算 | 使 DEEP-100M 可构图（基于距离的方法 OOM）|
| **Reverse edge addition** | 创建反向图，合并后各取 d/2 条边 | 减少强连通分量，增加 2-hop 覆盖 |

#### 搜索算法（四步循环）

```
① 随机采样 p×d 个节点，计算距离
② 归并排序候选缓冲区内 Top-M 列表
③ 图遍历：从未展开的父节点中选择邻居
④ 首次出现在候选列表中的节点 → 计算距离 → 回到②
```

**GPU 特有优化**：

- **Warp splitting**：32 线程的 warp 分为小组（低维 4-8 线程，高维如 GIST 用整组 32），每组加载一个向量元素块
- **Forgettable hash table**：共享内存中的"可遗忘哈希表"（2^8-2^13 条目），每 1-4 轮重置一次——**避免设备内存查找**
- **1-bit parented 管理**：用 MSB 标记父节点状态，避免哈希表查找，限制数据集 ≤ 2^31-1
- **Top-M 计算**：候选 ≤512 时用单 warp bitonic sort，更大时用 CTA 内 radix sort

#### 性能数据

| 指标 | vs HNSW（CPU） | vs GPU SOTA（GGNN/GANNS） |
|------|--------------|--------------------------|
| 构图时间 | **2.2–27× 更快** | 1.0–6.1× 更快 |
| 大批量 QPS（90-95% recall） | **33–77× 更快** | 3.8–8.8× 更快 |
| 单查询 QPS（95% recall） | **3.4–53× 更快** | — |

**生产部署案例（微信）**：
- 索引构建：小时级→分钟级（HNSW 的 1/30）
- 搜索吞吐：+50%+ QPS @ 同等 recall
- 基础设施成本：**-70%**

> **⚠️ 局限**：CAGRA 要求全索引驻留 GPU VRAM，十亿级需配合量化或跨卡；大批量性能远优于单查询
>
> **来源**：Ootomo et al., ICDE 2024 | [NVIDIA CAGRA 文档](https://developer.nvidia.com/rag/cagra)

**价值与代价**：
- ✅ **价值**：**构图速度量化标杆**——10M 1024D：45s vs HNSW 22min（**~30×**）；635M 1024D：56 min（8×H100）vs 6.22 天 CPU（**~160×**）；微信生产部署验证基础设施成本降低 **~70%**；33-77× 大批量（batch≥100）QPS 优势在稠密高维数据上稳定复现
- ⚠️ **代价**：**全索引驻留 GPU VRAM 使单卡支持上限约 21M（768D FP16）**——超过后必须量化或跨卡，引入额外 10-30% 性能损失；单查询（batch=1）优势降至 3.4-9×、在低维数据上可能仅 1.5-2×；构图显存开销约为索引大小的 **1.5-2×**（初始 k-NN 的高出度），需要 24GB A10G 以上 GPU；动态数据集（日更新 >5%）频繁重构图使 GPU 资源占用率从 30% 升至 70-90%

**深度洞察**：CAGRA 的设计选择蕴含了一个关于 GPU 向量搜索的深刻事实：**固定出度 + 单层图结构 + 随机采样起点**——这些看似与 HNSW 的"多层 + 可变出度 + 贪心下探"背道而驰的设计，实际上是 GPU 架构对算法施加的"暴力约束"——GPU 无法高效处理不规则的数据结构和控制流，所以 CAGRA 牺牲了 HNSW 的算法优雅性来换取 GPU 的执行效率。微信部署的 -70% 基础设施成本不是来自算法更好，而是来自 GPU 利用率更高。但 CAGRA 有一个没有被充分讨论的代价：**构图过程本身消耗大量 GPU 显存和计算资源**（两阶段：初始 k-NN + 剪枝），对于需要频繁更新索引的动态数据集，频繁重构图带来的 GPU 资源占用可能抵消其在搜索阶段的性能收益。**CAGRA 的成功前提是"静态数据集 + 大批量查询"——偏离这个前提时优势显著缩水。**

### 1.3 NVIDIA cuVS —— 新一代 GPU 向量搜索库

cuVS 是 RAPIDS RAFT 的继任者，自 cuVS 24.10（2024.10）起所有 ANN 和聚类算法从 RAFT 迁移至 cuVS。

#### 关键更新

#### 核心能力

**CAGRA-Q（量化 CAGRA）**：
- 在 GPU 显存中存储压缩向量，同时保持接近原始 recall
- 支持 FP8/INT8/INT4 精度

**多语言 API**：

**价值与代价**：

### 1.4 FAISS GPU —— CUDA 内核架构深度解析

FAISS GPU 不是简单的"在 GPU 上跑 FAISS"，而是包含多套专为 GPU 定制设计的 CUDA 内核。

#### 核心 CUDA 内核

| 内核 | 功能 | 加速阶段 |
|------|------|---------|
| `l2selectmin` | **融合内核**：IVF 质心距离计算 + k-selection 一步完成 | ③粗量器搜索 |
| `pqScanPrecomputedMultiPass` | PQ 查表扫描（预计算码本）| ④PQ 距离 |
| `runPQScanMultiPassNoPrecomputed` | PQ 扫描（无预计算）| ④PQ 距离 |
| `runIVFFlatScan` | IVF Flat 距离计算 + 选择 | ④Flat 距离 |
| `WarpSelect` | **Warp 级 k-selection** | ⑤Top-K 排序 |

#### WarpSelect 算法

FAISS GPU 无法使用优先级队列（无每核 SRAM），改用 **WarpSelect**：

```
① 每个 warp（32线程）从全局内存读取 32 个元素
② 每个 lane 维护一个**线程队列**（有序，大小 t）
③ 共享 warp 队列存储全局最小的 k 个值
④ 新值 < warp 队列最大时 → Batcher bitonic 排序合并
   → 线程队列大小：k≤32: t=2; k≤128: t=3; k≤256: t=4; k≤1024: t=8
```

**两段选择**：nprobe > 8 时，先降维到每个 probe ≤8 候选，再合并。

#### IVF-PQ 距离分解

```
d = ||x - y_C - y_R||² = ||x - y_C||² + ||y_R||² + 2⟨y_C, y_R⟩ - 2⟨x, y_R⟩
  其中：y_C=粗量化器中心，y_R=残差
```

- Term 1：粗量器阶段已计算
- Term 2：不涉及查询，可预计算
- Term 3：通过 PQ 查表近似
- Term 4：查询子向量 × 残差子向量质心（cuBLAS 批量矩阵乘）

#### 查询分片（Query Tiling）

FAISS GPU 将批次查询分片以管理临时内存：
1. 计算 `sizePerQuery`（每查询临时内存需求）
2. 确定 `queryTileSize`（多少查询并发处理）
3. 最小 tile size = 8；最大 = 65536
4. 双缓冲（2 streams）重叠计算

#### 内存布局：Interleaved-by-32

- 32 个向量分在一组，每组中同一维度的 32 个值连续排列
- 非字长编码（如 SQ6）按位打包：`(32×6)/8 = 24 bytes/dim`
- 使全局内存访问合并（coalesced），带宽利用率从 ~30% 提升至 ~80%

> **来源**：[FAISS GPU DeepWiki](https://deepwiki.com/facebookresearch/faiss/6-gpu-acceleration) | [FAISS GPU + cuVS](https://github.com/facebookresearch/faiss/wiki/GPU-Faiss-with-cuVS) | [Interleaved 布局 PR#1566](https://github.com/facebookresearch/faiss/pull/1566)

**价值与代价**：
- ✅ **价值**：Interleaved-by-32 将带宽利用率从 ~30% 提升至 ~80%（**~2.7×**），是内存受限场景下性价比最高的优化之一（纯软件修改，无硬件成本）；WarpSelect 在 k≤1024 时达到 **>90% 的 GPU 理论吞吐**，且 k=32 时仅需 t=2 线程队列（寄存器友好）；FAISS GPU 的成熟度（8 年+迭代、**10K+ GitHub stars**、Meta 生产验证）使其成为 GPU 向量搜索不可绕过的基线
- ⚠️ **代价**：k>1024 时 WarpSelect 退化为 radix sort，性能下降 **30-50%**；Interleaved-by-32 在 SQ6/SQ4 等非字长编码下打包/解包额外消耗 **5-15% 计算资源**；总体带宽利用率仍仅 ~80%（对比 NVIDIA 官方 cuBLAS 的 ~95%），差距 **15%**

**深度洞察**：FAISS GPU 的 Interleaved-by-32 内存布局将带宽利用率从 ~30% 提升至 ~80%，这个看似不起眼的"工程细节"实际上揭示了 GPU 向量搜索的**核心矛盾**：向量搜索是内存带宽密集型任务，但 GPU 的内存系统是为计算密集型任务设计的（大量线程高延迟容忍 + 缓存友好）。Interleaved-by-32 本质上是在 GPU 上模拟 CPU 的 SIMD 向量化访问模式——用软件手段弥补 GPU 内存系统在向量搜索场景下的"设计错配"。WarpSelect 算法同样揭示了另一个矛盾：**GPU 擅长大量并行计算，但 Top-K 排序是典型的归约操作，天然不适合 GPU 的 SIMT 模型**——需要复杂的 warp shuffles 和 bitonic 排序网络来模拟一个在 CPU 上只需要一个优先级队列的逻辑。这些"补丁"工作良好，但说明了 GPU 并非向量搜索的天然最佳架构——这些优化是在让 GPU 做它不擅长的事。

### 1.5 Tensor Core 距离计算 —— 将 L2 映射为矩阵乘法

#### 核心思想

L2 距离公式可分解为矩阵乘-累加操作：

```
||x - y||² = ||x||² + ||y||² - 2⟨x, y⟩
```

内积 `⟨x, y⟩` 直接映射到 Tensor Core MMA（Matrix Multiply-Accumulate）指令。这意味着 **Tensor Core 的数千 TFLOPS 可用于加速距离计算**，而非只能做乘加运算。

#### FaSTED（Fast and Scalable Tensor Core Euclidean Distance）

- 首个将 FP64 欧氏距离表达为 MMA 操作的算法
- 分层数据重用：全局内存 → 共享内存 → 寄存器

#### QTT（Quantized Tensor Train）分解引擎

QTT 将向量分解为张量序列，使得查询只需要一次 GEMV（通过 Tensor Core）：

| 指标 | 数值 |
|------|------|
| **1B 向量 单 H100 80GB 查询** | **38ms**, **100% recall**, 66GB serving size |
| 500M 384D FP16 | 3.1ms p50 延迟, 98% recall |
| 有趣现象 | recall **随规模增大而提升**（400M→96%, 1B→100%），与传统 ANNS 相反 |

#### 精度支持（H100 第 4 代 Tensor Core）

FP16 / BF16 / FP8 / INT8 / INT4 — 精度允许时可用更低精度获更高吞吐。

> **来源**：[Gallet & Gowanlock, IEEE HiPC 2022](https://ieeexplore.ieee.org/document/9975180) | [Milvus Blackwell 分析](https://blog.milvus.io/ai-quick-reference/how-does-blackwell-gb200-nvl72-change-milvus-cluster-sizing)
> **注意**：QTT 向量搜索来自 Holonomx 公司的 INVARIAN 产品（2025 HN 讨论），非学术论文。38ms/1B/100% recall 数据源自该公司技术演示，未经独立第三方验证。

**价值与代价**：
- ✅ **价值**：L2→矩阵乘映射使 Tensor Core 可用于距离计算——H100 FP16 Tensor Core 的理论吞吐 **990 TFLOPS**，是 CUDA core 的 ~**3×**；QTT（Quantized Tensor Train）在 H100 上达到 **38ms / 1B 向量 / 100% recall**（serving size 66GB），是目前已知最快的单卡暴力搜索方案；QTT 的 recall 随规模递增（400M→96%, 1B→100%）挑战了"近似搜索在大规模下必然损失 recall"的领域假设
- ⚠️ **代价**：QTT 向量搜索技术来自 **Holonomx 创业公司的 INVARIAN 产品**（2025-2026 HN 讨论），**非学术论文**，代码未公开，技术细节有限；38ms@1B 仅在 **384D FP16 特定配置**下验证（Holonomx 演示数据），泛化至 768D/1024D 时延迟预估 ~80-150ms；Tensor Core MMA 优势在高维受限——寄存器压力导致 warp 级并行度下降 **30-50%**（对比低维场景）

**深度洞察**：Tensor Core 距离计算是对 GPU 向量搜索"算力过剩"论点的最有力反驳。L2 距离分解为 ||x||² + ||y||² - 2⟨x, y⟩ 之后，内积部分可以高效地在 Tensor Core 上执行——**这颠覆了"向量搜索是内存密集型、Tensor Core 无用"的朴素认知**。QTT 的"recall 随规模增大而提升"的反直觉现象更值得深挖：这意味着对于某些经过良好结构化的表示（张量序列），增加数据点数量不仅不会增加搜索难度，反而提供了更多的"锚点"来约束搜索空间。如果这个结果可在更大规模（1B+）和更多数据集上复现，它将**从根本上挑战 ANNS 领域的隐含假设——近似搜索在更大规模下必然牺牲 recall**。QTT 可能开启一条避开 PQ/IVF/HNSW 的新路径：用结构化表示（张量分解）替代随机投影，使距离计算天然加速。

### 1.6 多 GPU 扩展 —— Fantasy 与 Blackwell NVL72

#### Fantasy（UC Santa Cruz, arXiv:2512.02278）

使用 **GPUDirect Async（IBGDA）** 在多 GPU 间流水线化向量搜索：

| 组件 | 细节 |
|------|------|
| 分区策略 | K-means 将查询路由到相关分区 |
| 卡间互连 | **NVLink**（600 GB/s 节点内）+ **RDMA**（25 GB/s/GPU 跨节点）|
| 流水线 | 两微批处理：K-means 分配 ∥ 并行 CAGRA ∥ 结果合并 |
| 性能 | 16 GPU（2×8）、1536D FP16：每卡 CAGRA ~68.5ms，分发仅 ~3.67ms |

**关键洞察**：图搜索是 **内存带宽受限** 而非计算受限。全索引驻留 HBM 避免 PCIe/NVMe 瓶颈。

#### Blackwell GB200 NVL72 影响

- **72 GPU NVLink 域**，吞吐达 H100 的 **30×**
- 以前需 8-16 GPU 节点的十亿级部署，现在可 1-2 个 NVL72 机架完成
- 能效 vs H100 **25×**

#### Scaling 效率预期（参考 ThemisDB 设计）

| 指标 | 1 GPU | 8 GPUs |
|------|-------|--------|
| QPS | ~200K | ~1.6M |
| P99 延迟 | ~8ms | ~15ms |
| 扩展效率 | — | ~80% |

> **来源**：[Fantasy, arXiv:2512.02278](https://arxiv.org/abs/2512.02278) | [Milvus Blackwell Blog](https://blog.milvus.io/ai-quick-reference/how-does-blackwell-gb200-nvl72-change-milvus-cluster-sizing)

**深度洞察**：多 GPU 向量搜索的核心矛盾在于 **"内存分布 vs 索引结构"**：K-means 分区将查询路由到相关 GPU——这意味着每个 GPU 只存储部分索引。但图索引（如 HNSW/CAGRA）的遍历性天然需要跨分区的邻居访问。Fantasy 的 K-means 路由本质上是在**把图搜索问题转化为倒排搜索问题**——用分区来规避 GPU 间的随机访问开销。这是一个聪明的工程妥协，但代价是：如果查询向量的最近邻分布在多个分区的边界上，recall 会下降。Fantasy 没有公开讨论这个"边界效应"。Blackwell GB200 NVL72 的 30× H100 吞吐宣称值得审慎对待——这是 72 GPU vs 1 GPU 的对比，非单卡性能提升。更重要的趋势是：**NVL72 的 72 GPU NVLink 域使得"整个十亿级索引驻留在统一的 GPU 内存池中"成为可能**——这可能在不需分区路由的情况下避免边界效应，但代价是全部依赖 NVIDIA 的专有硬件生态。

### 1.7 GPU-CPU 混合方案 —— 构图+查询分离的工程实践

业界对"纯 GPU 查询"的态度日趋务实——**GPU 构图 + CPU 查询** 正成为主流范式。

#### 模式一：cuVS CAGRA → HNSW 转换

```
流程：GPU 构建 CAGRA 图 → adapt_for_cpu → 转换为 HNSW → CPU 查询服务

效果：构图 12-15× 更快，CPU 查询更省钱
```

- Milvus 2.4 内置 `adapt_for_cpu` 参数
- GPU 构图 + CPU 查询的 **TCO 最低**

#### 模式二：Hot/Cold 分层调度（VectorLiteRAG, Georgia Tech）

向量访问存在**严重偏斜**——顶部 20% 的簇承载 **60-93%** 的访问。

```
粗量器路由 → 热簇（GPU, 20%）+ 冷簇（CPU SIMD, 80%）
            → 动态调度器 → LLM
```

- 延迟边界分区算法决定哪些簇放 GPU
- 以近 GPU 的延迟、近 CPU 的成本工作

#### 模式三：PQ 精炼（cuVS）

```
GPU IVF-PQ 粗搜（批量）→ CPU 用原始向量精炼重排序（小批量）
→ 接近精确 recall，GPU 吞吐优势 + CPU 原始向量无需搬运
```

#### 模式四：Elasticsearch ILM 分层

- **GPU 采集节点**（3 nodes, 64GB）：cuVS 构图，持有写入分片
- **CPU 热服务节点**（7 nodes, 192GB）：ILM rollover 后查询服务
- **BBQ 冷层**：8× 压缩（180 天以上数据）
- 结果：**12× 更快构图**，可扩展的 CPU 查询服务

> **来源**：[VectorLiteRAG, arXiv:2504.08930](https://arxiv.org/abs/2504.08930) | [Milvus GPU-CAGRA Hybrid](https://blog.milvus.io/zh/blog/faster-index-builds-and-scalable-queries-with-gpu-cagra-in-milvus.md) | [cuVS Blog](https://developer.nvidia.com/blog/enhancing-gpu-accelerated-vector-search-in-faiss-with-nvidia-cuvs/)

**深度洞察**：GPU-CPU 混合方案是"产业界的无声共识"——几乎每个向量数据库厂商的工程博客都在讨论 GPU 构图 + CPU 查询，但没有一家公开表示"纯 GPU 向量搜索在大规模场景下不经济"。这形成了一个有趣的**信息不对称**：厂商的营销材料强调 GPU 加速的倍数优势（构图 160×、查询 49×），而他们的架构白皮书详细解释如何将 GPU 用作一次性构图工具（CAGRA→HNSW）或用作物化视图的热数据加速器（Hot/Cold 分层）。VectorLiteRAG 的"20% 的簇承载 60-93% 的访问"是这篇报告中最具实用价值的单个数据点——它直接意味着对于大多数工作负载，**一个能容纳 20% 热数据的 GPU（可能只需 16-32GB）就能获得绝大多数加速收益**。这个规律决定了 GPU 在向量搜索中的最佳角色：**不是全量加速器，而是关键路径加速器**。

### 1.8 GPU 内存管理 —— 超 VRAM 场景的策略

向量规模超过单卡 VRAM 是常态而非例外，以下是主要应对策略：

| 策略 | 方法 | 适用场景 | 效果 |
|------|------|---------|------|
| **IVF-PQ 压缩** | 4-5× 压缩 vs FP32 | 100M 向量 ~360GB→~54GB | 性价比最高 |
| **Unified Memory** | `cudaMallocManaged` 透明页迁移 | 轻度超限 | 有页错误开销 |
| **RMM 池分配** | `PoolMemoryResource` 复用 GPU 内存 | cuVS 集成场景 | 减少碎片 |
| **CAGRA→HNSW** | 构图后转换，释放 GPU 显存 | 服务场景 | GPU 不参与查询 |
| **SIVF 流式 IVF** | 片上插入/删除的冲突免锁分配 | 实时更新 | 4.07M 向量/s 摄入（12 GPU）|

**实际建议**：

```
数据集稍微 > VRAM → Unified Memory
数据集远 > VRAM  → IVF-PQ 压缩（4-5×）
大规模查询服务  → CAGRA + RMM pool + HNSW 转换
实时流式更新   → SIVF（Streaming IVF）
热/冷分层     → GPU 热簇 + CPU/SSD 冷簇
```

> **来源**：[FAISS Memory Spaces](https://deepwiki.com/facebookresearch/faiss/6-gpu-acceleration) | [SIVF, arXiv:2601.11808](https://arxiv.org/abs/2601.11808)

**深度洞察**：GPU 内存管理的所有策略——IVF-PQ 压缩、Unified Memory、RMM 池、CAGRA→HNSW、SIVF——本质上都是在争取同一个东西：**把 GPU 有限的 HBM 用在最关键的地方**。这些策略的存在本身就是对 GPU 架构在向量搜索场景下的局限性的承认：**GPU 的 HBM 容量增长速度（H100→B200：80GB→192GB，2.4×）远低于企业向量数据的增长速度（每年 2-3×）**。Unified Memory 的自动页迁移看似最优雅，但在向量搜索场景下效率极低——向量搜索的随机访问模式会导致频繁的页错误和迁移抖动。SIVF 的 4.07M 向量/s 摄入吞吐在 12 GPU 上实现，意味着每秒需处理 ~14GB 的新数据——这虽然解决了流式更新问题，但也暗示了 GPU 在处理写入密集型向量工作负载时的高额开销。**一个容易被忽视的结论是：只要向量数据存储在 SSD 上、搜索时通过 PCIe 搬运到 GPU HBM，无论压缩率有多高，数据搬运的功耗和延迟都是不可避免的固定成本**——这是近存储计算方案（VSM、NDSEARCH）的根本机会。

### 2.1 ANNA（SNU, HPCA 2022）

**全称**：ANNA：Specialized Architecture for Approximate Nearest Neighbor Search（基于 Product Quantization 的专用加速器）

#### 加速定位

ANNA 专门针对 **PQ（Product Quantization）压缩向量** 的非对称距离计算（ADC）进行硬件加速，是 ASIC 路线中加速点最集中的方案之一。

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **④ 距离计算** | ✅ **核心设计目标** | 硬件实现 **非对称距离计算（ADC）**：将 PQ 码本查表（Look-Up Table）和加法树（Adder Tree）完全流水线化 |
| **⑦ 量化解码** | ✅ **主加速点** | PQ 码字表存储在片上 SRAM，每个子空间独立查表，消除量化瓶颈 |
| **⑤ Top-K 排序** | ✅ 硬件 Top-K 维护 | 专用排序单元，避免 CPU 的通用排序开销 |
| **③ 候选筛选** | ✅ ADC 结果筛选 | 芯片直接输出 Top-K 候选 ID，主机不需要做二次筛选 |
| **①② 索引构建/遍历** | ❌ | 依赖 CPU/GPU 完成，ANNA 仅做搜索 |

#### 关键架构特征

- **工艺**：TSMC 40nm GP @ 1GHz
- **面积**：17.51mm²
- **功耗**：仅 **5.398W**（ASIC 路线的能效标杆之一）
- **片上 SRAM**：存储 PQ 码本，避免片外访存
- **计算流水线**：查表（LUT）→ 累加（Adder Tree）→ Top-K 排序，全流水线无中断

#### 性能数据

| 指标 | 数值 |
|------|------|
| 吞吐改善 vs CPU（i7-7820X + Faiss/ScaNN） | **2.3–61.6×** |
| 延迟改善 vs CPU | **4.3–82.1×** |
| 数据集 | SIFT1M/1B, Deep1M/1B, GloVe |
| 验证状态 | ❌ 综合后仿真，未流片 |

> **⚠️ 注意**：GPU 基线为 V100（2017），未 vs H100/B200。论文发布于 2022 年，部分性能数据可能已落后于当前 SOTA。
>
> **来源**：[IEEE HPCA 2022](https://ieeexplore.ieee.org/document/9773206)

**价值与代价**：
- ✅ **价值**：5.398W / 17.51mm² 证明 ASIC 能以 GPU 1-2% 的功耗完成 PQ 搜索全流程；2.3-61.6× 的加速范围在大数据集（SIFT1B）上更接近下限 **~3×** 而非上限 **~60×**（小数据集），实际增益需按规模折半预期；片上 SRAM 存码本、全流水线 ADC 等架构决策被后续 10+ 篇论文引用
- ⚠️ **代价**：V100（2017）基线 → 到 H100（2022）GPU 带宽增长 **3.7×**，预估 ANNA 相对 H100 的加速比将降至 **2-8×** 而非报告的 61.6×；40nm 工艺 → 若迁移至 7nm 需全面重新综合，面积和功耗可能增加 **20-30%** 因模拟电路不随工艺线性缩放；综合后仿真的性能通常比流片实测高估 **15-30%**（时钟频率、功耗和温度影响未充分建模）

**深度洞察**：ANNA 的核心假设——"PQ 是 ANNS 的未来"——在 FP8/INT4 原生支持的 GPU 时代正受到挑战。当 GPU 可以直接用更高精度、更低开销的方式计算距离时，PQ 的跨子空间查表和加法树专用加速可能不再是必要的。ANNA 的"片上 SRAM 存码本"设计在 40nm 下有效，但在更先进工艺下 SRAM 密度收益递减，而 GPU HBM 带宽在代际间翻倍。更根本的问题是：**PQ 本身是否是一个过渡性技术？** 如果向量搜索的主流范式从"粗量化→精计算"转向"直接高精度匹配"（如 QTT 100% recall 1B 向量），那么为 PQ 定制的 ASIC 将成为针对一个过渡方案的专用硬件。

### 2.2 DReX（Duke, ISCA 2025）

**全称**：DReX: Memory Disaggregation for Vector Search

#### 加速定位

DReX 定位在 **内存分解（Memory Disaggregation）架构** 下的 **精确最近邻搜索**，通过两阶段流水线覆盖筛选和精确计算。

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **③ 候选筛选** | ✅ **第一级：PIM Filtering Units（PFU）** | 在 DRAM 内完成 **符号一致性过滤（Sign Concordance）**——用查询向量的符号位快速淘汰不可能成为近邻的向量。PFU 嵌入 DRAM bank 内，数据无需跨越 memory channel |
| **④ 距离计算** | ✅ **第二级：Near-Memory Accelerator（NMA）** | PFU 筛选后的候选向量进入近存加速器，执行 **精确 L2/Cosine 距离计算** |
| **⑥ 重排序** | ✅ 精确计算等价于重排序 | 本身是精确 NN 搜索，无需单独重排序 |
| **① 索引构建** | ❌ | CPU 完成 |
| **② 图遍历** | ❌ | 无图索引，完全暴力搜索 |
| **⑤ Top-K 排序** | ✅ | NMA 内置排序 |

#### 关键架构特征

- **互连**：CXL Type-3 设备（符合数据中心内存分解趋势）
- **内存**：LPDDR5X（高带宽、低功耗近存）
- **两级流水设计思想**：先用低成本符号过滤淘汰 **大部分** 非候选向量，再对剩余小部分做精确计算——**加速过滤阶段而不是加速计算阶段**

#### 性能数据

| 指标 | 数值 |
|------|------|
| CPU 对比（Faiss HNSW/IVF-SQ） | **24× / 19×**（batch 1/16） |
| GPU 对比（H100 CAGRA/ENNS） | **6.7× / 3.1×**（batch 1/16） |
| 精度 | Recall@32 = 0.95（Wiki 768D） |
| 验证状态 | ❌ 仿真（ISCA 2025 审稿人质疑基线可比性） |

> **来源**：[ACM ISCA 2025](https://dlnext.acm.org/doi/10.1145/3695053.3731079)

**价值与代价**：
- ✅ **价值**：DReX 的"先过滤再计算"两阶段设计（PFU 符号位过滤 + NMA 精确计算）是架构创新，其过滤思想可被其他路线借鉴；CXL Type-3 形态符合数据中心内存分解的大趋势；对比 H100（非 V100）是少数使用现代 GPU 基线的工作之一；PFU 的符号一致性过滤成本极低、无精度损失
- ⚠️ **代价**：ISCA 审稿人质疑基线对比公平性；依赖 CXL 3.0 生态大规模落地——**如果 CXL 内存分解未成为主流，DReX 将失去部署基础**；精确 NN 搜索场景有限（大多数企业可接受 95-99% recall），纯精确方案的差异化价值待验证；仿真阶段，未流片

**深度洞察**：DReX 的"先过滤再计算"原则（PFU→NMA）实际上揭示了一个通用的向量搜索优化哲学：**加速的真正杠杆不在于让距离计算更快，而在于减少需要计算距离的向量数量**。这与 SmartANNS 的"80% 距离减少"、VectorLiteRAG 的"20% 簇承载 93% 访问"背后是同一个逻辑。DReX 选择用符号位一致性作为过滤条件，这是一个**成本极低但精度可控**的设计——它隐含地承认了一个事实：对于大多数查询，大部分候选向量不需要被精确评估。这种"识别→跳过"模式比"全量→加速"模式更具扩展性。然而，DReX 对 CXL 生态的依赖是一个结构性的脆弱点——**如果 CXL 3.0 内存分解未能在数据中心大规模普及，DReX 将失去部署基础**，而这是学术论文通常不会纳入风险评估的维度。

### 2.3 IBM HERMES 芯片（Nature Electronics 2023）

**全称**：HERMES Project Chip——基于相变存储器（PCM）的模拟存内计算芯片

#### 加速定位

HERMES 并非专为向量检索设计，而是面向 **核方法（Kernel Methods）和注意力机制** 的通用模拟存内计算加速器。其核心加速的 **矩阵-向量乘法（MVM）** 是向量检索中距离计算的基础操作。

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **④ 距离计算（广义）** | ✅ **核心能力** | 在 PCM crossbar 阵列上以 **模拟方式** 完成 MVM，适用于核岭分类和核注意力中的距离/相似度计算 |
| **②/③ 搜索相关** | ❌ 非直接 | 不直接加速图遍历或候选筛选，适合小批量、高维核计算场景 |
| **精度** | ⚠️ 有效精度 ~3–4 bit | 核岭分类 <1% 精度损失；LRA 注意力仅差 0.08pp vs FP32 |

#### 关键架构特征

- **工艺**：14nm CMOS 混合信号
- **架构**：64 核，每核 256×256 PCM crossbar 阵列
- **峰值吞吐**：**63.1 TOPS** @ **6.5W**（9.76 TOPS/W）
- **工作方式**：数据保留在 crossbar 阵列中，无需从片外搬运权重，**消除内存墙**

#### 性能数据

| 指标 | 数值 |
|------|------|
| 峰值吞吐 | **63.1 TOPS** |
| 能效 | **9.76 TOPS/W** |
| 峰值功耗 | **6.5W** |
| 有效精度 | ~3-4 bit |
| 核分类精度差 | **<1%** vs FP32 |
| LRA 注意力精度差 | **0.08 pp** vs FP32 |
| 验证状态 | ✅ **已流片并实测**（14nm 真实芯片）|

> **来源**：[arXiv:2411.03375](https://arxiv.org/pdf/2411.03375)（Nature Electronics 2023）

**价值与代价**：
- ✅ **价值**：**全文唯一已流片且实测的 AIMC 芯片**（Nature Electronics 同行评审），6.5W / 63.1 TOPS（9.76 TOPS/W）能效是 H100（~0.7 TOPS/W）的 **~14×**；14nm 量产工艺，64 核 × 256×256 crossbar；在高斯核分类和 LRA 注意力基准上与 FP32 精度差 <1%（**0.08 pp**），证明模拟计算可在实际任务中匹配浮点精度
- ⚠️ **代价**：**~3-4 bit 有效精度**（等效于 8-16 级量化）→ 在需 >99% recall 的高精度向量搜索中，混合精度方案（模拟粗算+数字精修）将使系统级功耗增加 **2-3×**，能效优势从 ~14× 降至 ~5×；芯片为研究原型而非产品，距商业化至少 **2-3 年**；温度敏感性和 PCM 写耐久性（典型 PCM ~10^8 次擦写）等工程问题未充分讨论

**深度洞察**：HERMES 是全文唯一已流片的模拟存内计算芯片，但其真正的价值信号不在性能数据里——6.5W / 63.1 TOPS 的数字固然亮眼，但~3-4 bit 的有效精度意味着它在向量检索场景中面临一个**"精度税"困境**：如果要匹配 FP32 的召回率，要么需要混合精度方案（模拟粗算 + 数字精修）增加系统复杂度和功耗，要么接受 recall 损失。对于需要高召回率的搜索场景（如企业 RAG 的知识检索漏检代价极高），这个精度税可能抵消掉能效优势。HERMES 更适合的目标可能是**低精度高吞吐的近似任务**（如粗筛、嵌入生成的第一阶段），而非高精度向量搜索。更深层的问题是：**模拟存内计算的精度-功耗 Pareto 前沿是否能在合理时间内推进到 FP16 等效？** 如果答案是否定的，HERMES 将成为"极致能效但精度受限"的 niche solution，而非向量搜索的主流加速选项。

### 2.4 IBM Spyre AI 加速器

#### 加速定位

Spyre 是 IBM 的 AI 推理 ASIC，在向量检索场景中承担 **精确暴力搜索加速** 和 **嵌入生成加速** 两个角色。

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **④⑥ 精确暴力搜索** | ✅ **核心场景** | 12 卡 Spyre 服务器在 **数秒内** 对 **10 亿向量（384 维）** 完成 **100% recall 的暴力搜索** |
| **⑦ 嵌入生成** | ✅ **多用途** | 运行 Granite-Embedding-125M，吞吐在 L40S 与 H100 **之间** |
| **① 索引构建** | ❌ 非适用 | 精确搜索无需索引 |
| **②-⑤ ANN 搜索** | ❌ 非适用 | 定位为 ANN 的补充（双轨策略中的"精确轨"） |

#### 在 IBM CAS 中的角色

```
常规查询：ANN 搜索（694ms，90% recall，存储层索引）
    ↓ 当需要 100% recall 时
精确查询 → IBM Spyre 暴力搜索（数秒，100% recall）
```

#### 关键指标

| 指标 | 数值 |
|------|------|
| 嵌入吞吐定位 | 在 NVIDIA L40S 和 H100 **之间** |
| 能效比 | 每瓦吞吐 **8.8× vs H100** |
| 精确搜索能力 | 12 卡 → 10 亿 384D 向量 / 数秒 / 100% recall |

> **⚠️ 注意**：数据主要来源为 IBM 社区博客和工程报告，非独立第三方评测。
>
> **来源**：[IBM Community Blog](https://community.ibm.com/community/user/blogs/henrik-mader/2026/05/13/spyreandembeddingmodels)

**价值与代价**：
- ✅ **价值**：**100% recall 精确搜索填补 ANN 无法覆盖的场景**——12 卡数秒 / 10 亿 384D 向量，是已知最快的可部署精确搜索方案之一；嵌入场景每瓦性能 **8.8× vs H100**（Granite-Embedding-125M），单卡吞吐位于 L40S 和 H100 **之间**
- ⚠️ **代价**：**查询延迟秒级**（预估 2-5s）vs GPU/FPGA 毫秒级 → 慢 **200-1000×**，仅可用作安全网；8.8× 嵌入能效 ≠ 向量搜索能效（Spyre 在带宽受限的搜索中优势可能 <**3×**）；12 卡服务器（预估 $50-80K）部署成本 + 运维复杂度；数据仅来自 IBM 社区博客，**无独立第三方评测**

**深度洞察**：Spyre 在报告中的角色具有战略意义——它揭示了 IBM 的"双轨策略"：ANN（CAS 的 694ms / 90% recall）处理常规查询，Spyre（数秒 / 100% recall）处理需要精确结果的场景。这不是技术选择，而是**产品定位选择**——IBM 押注的企业场景中，某些领域（金融合规、医疗诊断、法律检索）对 recall 的要求是 100%，任何近似方案都无法满足。Spyre 的 8.8× 每瓦性能 vs H100 值得关注，但需要认识到这个比较是基于**嵌入生成**而非向量搜索——嵌入是矩阵乘密集型（适合 Spyre 架构），而向量搜索是内存带宽密集型（Spyre 在此场景的优势可能远小于 8.8×）。12 卡服务器数秒完成 10 亿向量精确搜索，意味着查询延迟在**秒级**而非毫秒级——这决定了 Spyre 定位为"安全网"而非"主搜索路径"。

### 2.5 NDSEARCH（Duke/Samsung, ISCA 2024）

**全称**：NDSEARCH: Accelerating Graph-Traversal-Based ANNS through Near Data Processing

#### 加速定位

NDSEARCH 在 **NAND 闪存芯片内部** 集成 LUN 级近数据处理加速器，目标是加速 **基于图遍历的 ANN 搜索** 中的 **数据访问和距离计算**。

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **④ 距离计算** | ✅ **核心加速** | LUN 级加速器在数据从 NAND 读出的同时完成距离计算，计算与 I/O 重叠 |
| **② 图遍历** | ✅ **关键优化** | NDP 加速器处理图遍历中的邻居访问和候选距离计算，减少主机参与的遍历次数 |
| **②/③ 候选剪枝** | ✅ | 利用 SSD 内并行性（跨 LUN、跨 channel）加速候选探索和剪枝 |
| **⑤ Top-K 排序** | ⚠️ 部分 | 加速器协助维护距离列表，但 Top-K 合并仍涉及主机 |
| **① 索引构建** | ❌ | CPU 完成 |
| **⑥ 重排序** | — | 主要用于 ANN，不单独重排序 |

#### 关键技术创新

- **LUN 级粒度**：比 Chip-level（DeepStore）更细的并行度，**2.9× 超越 DeepStore**
- **SSD 内部流水线**：充分利用 SSD 内部的多通道、多芯片、多 plane 并行性
- **计算存储一体化**：消除 PCIe 搬运开销

#### 性能数据

| 指标 | 数值 |
|------|------|
| 吞吐改善 vs CPU | **31.7×** |
| 吞吐改善 vs GPU | **14.6×** |
| 吞吐改善 vs SmartSSD 仅方案 | **7.4×** |
| 能效 vs CPU | **178.68×** |
| 能效 vs GPU | **120.87×** |
| 系统功耗 | **26.32W**（PCIe 55W 预算内） |
| 存储开销 | ~6% 存储密度损失 |
| 验证状态 | ❌ 仿真（SSD-Sim + Synopsys 32nm） |

> **来源**：[ISCA 2024](https://doi.org/10.1109/ISCA59077.2024.00035)

**价值与代价**：
- ✅ **价值**：**全文能效比最高**——仿真报告 178× CPU / 120× GPU，系统功耗仅 **26.32W**（PCIe 55W 预算内）；LUN 级并行度相比 chip-level（DeepStore）额外提升 **2.9×**，证明 SSD 内细粒度加速器方向可行
- ⚠️ **代价**：**178× 基于 cycle-accurate 仿真而非实测**——仿真到流片的典型性能衰减 2-5×，预估真实芯片能效可能在 **30-60×** 范围；LUN 级集成需三星/铠侠等 NAND 原厂修改闪存晶圆设计和测试流程，工程投入 **$50-100M+**，量产周期 **3-5 年**；6% 存储密度损失 → 对 1TB SSD 等价于损失 **60GB** 可用容量，按 NAND 价格 ~$0.05/GB 计算每 SSD 增加成本 **~$3**，在千亿级数据中心部署中累计代价显著

**深度洞察**：NDSEARCH 的 178.68× CPU 能效是全文最高数字，但也是最需要谨慎解读的数字——这是基于 Synopsys 32nm 工艺仿真的 RTL 结果，而非真实芯片测量。该方案真正的洞察价值在于其**并行粒度选择**：LUN 级（SSD 内部最细粒度的独立操作单元）比 Chip-level（DeepStore）更细，能利用更多内部并行度，但代价是存储密度损失 ~6%。这个取舍揭示了一个根本问题：**在 NAND 内集成计算的边际收益在递减**——从 chip-level 到 LUN-level 的收益只有 2.9×，而每一步更细粒度的集成都会带来存储密度、良率和测试成本的惩罚。NDSEARCH 可能已经接近了"在 NAND 内加计算"这一方向的收益上限。更深层的问题是：**如果 GPU HBM 带宽持续翻倍、CXL 内存池化成为主流，SSD 内计算的价值定位是否会从"主搜索路径"退化为"冷数据搜索的兜底方案"？**

### 2.6 Cosmos（IEEE CAL 2025）

**全称**：Cosmos: A CXL-Based Full In-Memory System for Approximate Nearest Neighbor Search

#### 加速定位

Cosmos 在 CXL 内存设备中集成通用核心，将 ANNS 搜索 **完全卸载** 到 CXL 内存端。

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **④ 距离计算** | ✅ **Rank 级并行** | 将向量维度拆分到 CXL 设备的多个 rank 中，每个 rank 并行计算部分距离 |
| **② 数据布局** | ✅ **邻域感知放置** | 将邻近向量放置在同一 CXL 设备内，减少跨设备访问 |
| **③ 搜索卸载** | ✅ | 主机只发送查询，CXL 设备完成全部搜索流程 |
| **① 索引构建** | ❌ | 主机完成 |

#### 性能数据

- **6.72×** 吞吐 vs 基线 CXL 系统
- **2.35×** vs 前代 CXL 方案（Lupus）

> **来源**：[IEEE CAL 2025](https://ieeexplore.ieee.org/document/11004422)

**价值与代价**：
- ✅ **价值**：CXL rank 级并行 ANNS 卸载（6.72× vs 基线 CXL、2.35× vs 前代 Lupus），开创"将 ANNS 作为 CXL 原生负载"的方向；邻域感知数据布局设计可迁移至 NUMA、多 GPU 架构
- ⚠️ **代价**：**性能对比仅限 CXL 生态内部**——未与 H100 CAGRA 或 CPU HNSW 对比；CXL 3.0 生态预计 **2027-2028 年**才进入大规模部署，比论文发表晚 2-3 年，存在"论文发布时生态未就绪"的风险；IEEE CAL 短文限制（6 页），缺乏完整微架构和实现细节

**深度洞察**：Cosmos 和 DReX 代表了同一种趋势——**CXL 生态下的存算融合**。但它们有一个共同的隐性假设：CXL 3.0 内存分解将在数据中心大规模普及。这个假设尚未被验证。CXL 面临"鸡和蛋"的问题：没有杀手级应用 → 部署量不足 → 价格下不来 → 没有杀手级应用。Cosmos 的 6.72× vs 基线 CXL 系统的性能提升在此背景下意义有限——如果整个 CXL 生态没有起飞，相对于自己的提升没有参照价值。Cosmos 真正的贡献是 **"邻域感知数据布局"**——将邻近向量放置在同一 CXL 设备内以减少跨设备访问——这个设计原则在 NUMA、多 GPU 等其他架构中同样适用，其价值可能超越 CXL 本身。

### 2.7 ANSMET（清华大学, ISCA 2025）

**全称**：ANSMET: Approximate Nearest Neighbor Search with Near-Memory Processing and Hybrid Early Termination

#### 加速定位

ANSMET 在 **DRAM DIMM 缓冲芯片** 中嵌入近数据处理单元，让距离计算在内存等级上完成，同时用 **混合粒度早停（Hybrid Early Termination）** 大幅减少无效数据读取。

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **④ 距离计算** | ✅ **DIMM 级 NDP 核心** | 在 LRDIMM 的每个数据缓冲器（DB）中放置距离计算单元（16 加法器 + 16 乘法器），计算部分距离后通过片间总线发送至中央 RCD 汇总 |
| **③ 候选筛选** | ✅ **早停（Early Termination）** | 在向量数据逐渐获取过程中，用部分维度 + 部分位的保守下界估计来判断是否提前终止该向量的距离计算——**无需精度损失** |
| **② 图遍历** | ⚠️ 主机 CPU | 主机 CPU 负责 HNSW 图导航，将比较任务卸载到 NDP 单元 |
| **⑤ Top-K 排序** | ❌ 主机 CPU | 主机 CPU 完成 |
| **① 索引构建** | ❌ 主机 CPU | 主机 CPU 完成 |

#### 核心技术创新

**混合早停（Hybrid Early Termination）**——ANSMET 最具原创性的贡献：

```
传统方法：完整获取向量 → 完整计算距离 → 比较 → 决定保留/丢弃
           ↑ 大量 DRAM 带宽浪费于最终被淘汰的向量

ANSMET 方法：
 ① 部分维度获取 → 保守估计距离下界 → 若下界 > 阈值 → 早停（无需更多获取）
 ② 部分位获取（最高有效位优先）→ 继续细化估计 → 若下界 > 阈值 → 早停
 ③ 完整获取 + 精确计算 → 更新阈值
```

- **维度级 ET**：获取部分维度，假设未获取维度取"使距离最小化"的值来计算下界
- **位级 ET**（论文新颖贡献）：在每个维度内，从 MSB 开始逐块读取，用部分位估计下界
- **混合策略**：粗粒度位块（`nc`）快速扫描 + 细粒度位块（`nF`）精确终止，通过采样确定最优参数

**离线公共前缀消除**：
- 识别所有或大多数向量元素共有的高位前缀 → 从存储中省略 → 单副本保存在片上 QSHR
- 异常向量用每向量/每元素元数据位标记 → 相当于一种针对 ANNS 数据布局的定向量化

**热向量复制**：
- 利用 HNSW 索引结果的可预测性（顶层顶点访问频繁）→ 跨等级复制热向量 → 改善负载均衡
- 这是 ANNS 索引结构相比推荐系统中任意访问模式的独特优势

#### 关键架构特征

| 维度 | 数据 |
|------|------|
| **技术路线** | ASIC（DIMM 缓冲芯片内 NDP 单元）|
| **NDP 单元数量** | 32 个（4 通道 × 2 DIMM × 4 等级），每单元 1.2 GHz |
| **每单元面积** | **0.06 mm²**（22nm 评估，可忽略不计）|
| **片上 SRAM（QSHR）** | 每单元 67.125 kB（存储查询向量 + 比较任务状态）|
| **互连** | DDR5-4800，片间分层数据总线 |
| **距离度量** | L2/欧几里得、内积、余弦 |
| **索引支持** | HNSW（主要评估）、IVF/树等也可兼容 |
| **精度** | **无精度损失**（保守下界保证不误淘汰真实近邻）|
| **开源** | [github.com/tsinghua-ideal/ANSMET](https://github.com/tsinghua-ideal/ANSMET) |

#### 性能数据

| 指标 | 数值 |
|------|------|
| NDP-Base vs CPU 基线 | **平均 5.26× 加速**（DEEP 数据集最高 6.40×）|
| NDP + 完整 ET vs CPU 基线 | **~8× 加速**（NDP-Base 基础上额外 1.52×）|
| 能效 vs CPU 基线 | **~20% 能耗**（节能 77.8% + ET 额外贡献 18.3%）|
| DRAM 获取利用率 | CPU 基线 6.0% → ANSMET **11.1%**（仍有优化空间）|
| 验证状态 | ❌ Cycle-accurate 模拟（Ramulator 2.0 + CACTI 22nm）|
| 作者 | 清华大学 李一苇、金宇鑫、田博宇、张焕晨、高明宇 |

> **来源**：[ACM ISCA 2025](https://dl.acm.org/doi/10.1145/3695053.3731013) | [GitHub](https://github.com/tsinghua-ideal/ANSMET)

**价值与代价**：
- ✅ **价值**：混合早停（维度和位级）实现 **~8× CPU 加速**（NDP-Base 5.26× + ET 额外 1.52×），**无精度损失**——保守下界保证不误淘汰真实近邻；每 NDP 单元仅 **0.06mm²**（22nm），32 单元总面积 **<2mm²**，在 DIMM 缓冲芯片 ~100mm² 面积预算中可忽略；系统能耗降至 CPU 基线的 **~20%**（节能 77.8% + ET 额外 18.3%）；开源（GitHub）有助于复现
- ⚠️ **代价**：**仍 89% DRAM 获取无效**——早停仅将利用率从 6.0% 提至 11.1%，根本原因是向量搜索的随机访问模式无法被早停完全消除；cycle-accurate 模拟（Ramulator 2.0）与真实芯片差距通常 **15-30%**（DRAM 时序的物理效应难以精确建模）；采样优化（100 向量→10 亿参数）缺乏统计保证；静态离线预处理**不支持频繁更新**的数据集（日增 >1% 时需重新采样和布局）

**深度洞察**：ANSMET 的核心洞察——"**不要加速距离计算，要避免不必要的距离计算**"——是整个向量搜索硬件加速领域最被低估的设计原则。获取利用率从 6.0% 提升到 11.1% 意味着即使经过优化，仍有 ~89% 的 DRAM 数据获取是"无效"的（这些向量最终不会被选入 Top-K）。这个数字揭示了一个被所有方案回避的事实：**向量搜索的主导开销不是计算距离本身，而是把大量最终会被淘汰的向量数据从内存搬出来**。ANSMET 的混合早停本质上是"以少量计算（比较下界）换取大量 I/O（跳过完整获取）"的权衡——这恰恰是算术强度分析告诉我们的正确方向：在内存带宽受限的系统中，用计算换 I/O 是划算的。DReX 的 PFU 符号过滤和 ANSMET 的混合早停虽然实现完全不同（DRAM 内符号位比较 vs DIMM 缓冲芯片内渐进式计算），但背后的设计哲学完全一致。**它们的共同启示是：向量检索加速的下一步不应该是让距离计算更快，而是让更多向量根本不需要被完整计算**。

### 3.1 IBM VSM on FCM

**VSM（Vector Search Module）= FCM（FlashCore Module）硬件的向量搜索模式**

VSM 是 **FPGA 路线中加速覆盖最完整、性能最强** 的方案——在 SSD 控制器 FPGA 内部完成从 NAND 读取到 Top-K 结果输出的 **全流水线加速**。

#### 加速定位

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **③ 候选筛选** | ✅ **LUN 级并行读取** | 向量子簇跨 **所有 NAND channel/die 条带化**，最大化读取并行度；SLC 模式消除 ECC 延迟瓶颈 |
| **④ 距离计算** | ✅ **32 条并行硬件 MAC 流水线** | 数据从 NAND 读出后直接进入 **ECC 解码 → 距离乘加 → 排序** 的流水线，**不经 DRAM、不经 PCIe** |
| **⑤ Top-K 排序** | ✅ **硬件 KNN 排序引擎** | 基于最大堆（max-heap）的硬件排序，K=1 到 K=100 **延迟相同**（流水线开销不变） |
| **① 索引构建** | ❌ | 主机 CPU/GPU 完成 |
| **② 图遍历** | ❌ | VSM 做暴力搜索 + IVF 簇内扫描；图遍历由主机完成 |
| **⑥ 重排序** | ✅ VSM 暴力搜索本身就是精确计算 | 精度优于 ANN |

#### 算法-硬件协同设计细节

```
主机发送查询向量
    ↓
VSM (FPGA 内部流水线)：
  NAND Flash (32 parallel channels)
    → ECC Decode (SLC 模式简化)
    → 32× 并行 MAC 流水线 (Distance Computation)
    → 硬件 KNN 排序引擎 (Max-Heap)
    → 返回 Top-K 结果 (仅 K 个 ID，原始向量不跨 PCIe)
```

#### 与 HNSW/IVF 的配合

| 搜索策略 | VSM 角色 | 主机角色 |
|---------|---------|---------|
| **暴力搜索（高 recall）** | ✅ 全流水线在 FPGA 内完成 | 发送查询 + 接收结果 |
| **IVF 亿级搜索** | ✅ 簇内暴力扫描 | ✅ 索引导航（决定搜索哪些簇） |
| **HNSW 图搜索** | ❌ 不适配 | 由主机或另一个加速器完成 |

#### 性能指标

| 指标 | 数值 | 加速阶段 |
|------|------|---------|
| 暴力搜索延迟（1M 向量） | **<10ms** | **③+④+⑤ 全流水线** |
| 单 VSM 吞吐 | **>17,000 QPS** | ③+④+⑤ |
| 48× VSM 扩展 | **近线性** | ③ 跨设备并行 |
| 能效 vs 4× H100 GPU | **~30× 更优** | ④ 距离计算能效 |
| LPT 翻译表缩小 | **~3500×** | ③ 地址翻译加速 |
| 存储密度损失 | ~6%（因 SLC 模式） | 权衡，非直接加速 |

> **⚠️ 注意**：
> - VSM 需要 10k+ 并发查询才能饱和
> - IVF 的查询发散（跨簇 scatter）需要智能调度，否则流水线利用率下降
> - 技术细节主要来自 IBM 中文技术社区和 53ai 等非正式渠道，尚未见正式产品公告或顶级会议论文
>
> **来源**：IBM 中文技术社区；53ai.com 深度文章；相关学术工作 ChamVS 由 ETH Zurich 的 Wenqi Jiang 等人完成

**价值与代价**：
- ✅ **价值**：**双模设计（FCM↔VSM）解决硬件利用率问题**——数据中心 SSD 利用率通常 30-50%，VSM 模式将其转化为算力储备且不增加硬件采购成本；全流水线消除 PCIe 搬运，单 VSM >**17K QPS**、**<10ms @ 1M 暴力搜索**、**~30× 能效 vs 4× H100**；K 无关延迟（K=1=K=100）是硬件独有优势；**48× 近线性扩展**（效率 >90%）远超 GPU 多卡（60-80%）
- ⚠️ **代价**：**信息可信度全文最低**——数据来自 IBM 技术社区和 53ai，未经正式产品公告或同行评审；需 **10k+ 并发才能饱和**，企业级 RAG 场景（100-1000 QPS）利用率仅 **1-10%**，能效优势无法兑现；IVF 查询发散导致流水线利用率从 100% 降至 **30-50%**，需智能调度补偿；SLC 模式牺牲 **~50%** 存储密度（6%是相对特定 NAND 的估算），容量需求翻倍增加系统成本；公开信息不足以独立复现性能数据

**深度洞察**：VSM 真正的战略价值不在于其性能数据，而在于它的**双模设计**——同一块硬件可以切换为 FCM（标准块存储）或 VSM（向量搜索加速）。这在数据中心层面解决了一个被忽略的问题：存储硬件的利用率通常很低（~30-50%），而搜索加速器（如 GPU）的价格昂贵且供给有限。VSM 允许在存储硬件空闲时将其"借用"为搜索加速器，本质上是在**不增加硬件采购成本的前提下释放了算力储备**。3500× LPT 翻译表缩小是一个被低估的工程创新——它意味着 VSM 不需要额外的 DRAM 来管理地址翻译，这在 SSD 控制器有限的功耗/面积预算内是关键突破。同时，VSM 的 K 无关延迟（K=1 和 K=100 延迟相同）揭示了硬件排序引擎相对于软件排序的本质优势：**排序时间由流水线深度决定，不受 K 值影响**。但 VSM 的 10k+ 并发要求是一个实际部署门槛——对于低吞吐场景（如企业级 RAG 的分钟级查询），VSM 将处于严重欠载状态，其能效优势可能无法兑现。

### 3.2 Falcon（ETH Zurich, VLDB 2025）

**全称**：Fast Graph Vector Search via Hardware Acceleration and Delayed-Synchronization Traversal

#### 加速定位

Falcon 是 **FPGA 上基于图的向量搜索加速器**，其独特贡献在于 **算法-硬件协同设计**——DSL 遍历算法专门为 FPGA 的有限片上资源定制。

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **② 图遍历** | ✅ **核心设计目标** | **Directed Sonar Traversal（DST）**——一种延迟同步的图遍历算法，比经典 BFS **快 1.7–2.9×**。DST 降低了对片上 SRAM 的同步需求，适配 FPGA 资源约束 |
| **④ 距离计算** | ✅ **FPGA 流水线** | 片上流水线完成遍历过程中的所有距离计算，避免 CPU 的指令缓存/分支预测开销 |
| **③ 候选筛选** | ✅ DST 的定向遍历天然起到剪枝作用 | R@10 提升 **0.14–4.93%**（DSL 比 BFS 更能找到相关邻居） |
| **① 索引构建** | ❌ | 使用 CPU 构建 HNSW/NSG 图 |
| **⑤ Top-K 排序** | ⚠️ | 依赖主机或简单硬件维护 |

#### DST 算法的硬件动机

- **经典 BFS 的问题**：每步扩展需要大量同步 → 频繁的片上-片外数据交换
- **DST 的解决方案**：延迟同步，允许"过时"的候选向量继续遍历，减少同步次数
- **FPGA 适配**：减少同步 → 减少对片上 BRAM/URAM 容量的需求 → 可以在 Alveo U250 上容纳 10M 规模

#### 性能数据

| 指标 | 数值 | 加速阶段 |
|------|------|---------|
| CPU 延迟比（Xeon 8259CL） | **4.3×** ↓ | **②+④ 遍历+距离** |
| GPU 延迟比（V100） | **19.5×** ↓ | **②+④ 遍历+距离** |
| CPU 能效比 | **8.0×** ↑ | ④ 距离计算能效 |
| GPU 能效比 | **26.9×** ↑ | ④ 距离计算能效 |
| DST vs BFS 加速 | **1.7–2.9×** | **② 遍历算法** |
| 功耗 | 55–62W | — |

#### 局限性

- **10M 规模上限**（受限于 FPGA 片上内存）
- **GPU（H100）在大 batch 下反超**（12× 内存带宽优势）
- GPU 基线为 V100（2017），现代 H100/B200 差距会缩小

> **验证**：✅ 3-0 全票通过
>
> **来源**：[Jiang et al., PVLDB 2025](https://dl.acm.org/doi/abs/10.14778/3749646.3749655)

**价值与代价**：
- ✅ **价值**：DST 算法通用贡献——延迟同步减少贪心遍历中的片上-片外交换，**1.7-2.9× vs BFS** 且 R@10 提升 **0.14-4.93%**；FPGA 原型（AMD Alveo U250）**物理实测**，55-62W 功耗远低于 GPU（V100 183-324W）；论文完全公开算法、架构和实验配置，可复现性好
- ⚠️ **代价**：**GPU 基线 V100（2017）**——到 H100（2022）GPU 带宽提升 3.7×，26.9× 能效优势预估缩至 **~7×**，B200（2024）后可能仅 **3-5×**；**10M 规模上限**（FPGA 片上 BRAM/URAM 限制）→ 最需要加速的十亿级场景用不上；大 batch 下 GPU 以 **12× 带宽优势** 反超——Falcon 在小 batch（<100）有优势，大 batch（≥1000）回归 GPU

**深度洞察**：Falcon 的核心贡献其实是 DST 算法，而非 FPGA 实现本身。DST 的"延迟同步"思想——允许过时的候选向量继续遍历——可能对通用 GPU 向量搜索也有启发意义。如果 DST 被移植到 GPU（cuVS 的 CAGRA 遍历算法中引入类似机制），Falcon 的 FPGA 优势将消失，但 DST 作为算法的价值会放大。这是学术界的常见模式：**一个为特定硬件设计的技术（DST-for-FPGA），如果其核心思想具有通用性，其长期影响力可能让硬件原型本身过时**。Falcon 的 10M 规模限制（受 FPGA 片上内存约束）不是算法限制而是硬件限制——这恰恰说明 FPGA 不适合大规模图搜索（图需要大量随机访问内存），而更适合流水线化计算。**Falcon 对比 GPU 的 26.9× 能效提升本质上不是 FPGA 的胜利，而是 DST 算法减少了同步开销的结果**——如果同一算法在 GPU 上实现，部分优势可能保留。

### 3.3 FANNS（ETH Zurich, 2023）

**全称**：FANNS: FPGA-based Approximate Nearest Neighbor Search（端到端 FPGA IVF-PQ 框架）

#### 加速定位

FANNS 是 **IVF-PQ 方法在 FPGA 上的完整实现**，自动协同设计硬件和算法满足用户指定的召回目标和资源预算。

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **③ IVF 扫描** | ✅ **被动扫描加速** | FPGA 流水线并行扫描多个倒排链，链式距离计算无停顿 |
| **④ 距离计算** | ✅ **PQ 非对称距离** | FPGA 硬件实现 PQ 码本查表 + 部分距离求和，全流水线 |
| **⑦ 量化解码** | ✅ **查表加速** | PQ 码本存储在 FPGA BRAM，并行子空间查表 |
| **⑤ Top-K 排序** | ✅ 硬件排序器 | FPGA 上专用 Top-K 模块 |
| **① 索引构建** | ❌ | CPU 完成 IVFPQ 训练和索引 |
| **② 图遍历** | ❌ | IVFPQ 无图遍历 |

#### 性能数据

| 指标 | 数值 |
|------|------|
| 吞吐 vs 基线 FPGA | **20.79×** |
| 吞吐 vs CPU | **29.98×** |
| 硬件 | AMD Alveo U55c FPGA |
| 局限 | GPU（V100）仍优于 FANNS |

> **来源**：[ETH Zurich SPCL](https://spcl.inf.ethz.ch/Publications/index.php?pub=496)

**价值与代价**：
- ✅ **价值**："自动协同设计"理念前瞻——给定 recall 目标和硬件预算自动生成 FPGA 加速器，将 FPGA 开发周期从 **数月缩短至数天**；IVF-PQ 全流水线在 FPGA（Alveo U55c）上实现了 **29.98× 吞吐 vs CPU**（基线 FPGA 对比的 20.79×），证明 FPGA 可承载完整的 ANN 搜索流程
- ⚠️ **代价**：**20.79× 是对比自己的基线 FPGA 而非 CPU/GPU**，参照系价值有限；论文自身承认 **"GPU（V100）仍优于 FANNS"**——FPGA 全流水线加速后仍不如通用 GPU；对比 GPU 时未报告具体加速比（暗示 <1×），说明 GPU 的通用计算对 IVF-PQ 已足够高效

**深度洞察**：FANNS 最重要的贡献不是性能数字（20.79× vs 基线 FPGA 是一个"自己比自己"的指标），而是 **"自动协同设计"的理念**——给定 recall 目标和硬件预算，自动生成对应的 FPGA 加速器。这指向了 FPGA 加速器设计的未来方向：**硬件即编译器输出**（Hardware as a compiler target）。如果这一理念成熟，FPGA 在向量搜索领域的角色将从"手写 RTL 加速"变为"编译产出的加速器"，大幅降低开发成本。但 FANNS 的"GPU 仍优于 FANNS"这一结论值得深思——**在 FPGA 上实现了全部的 IVF-PQ 流水线加速后，仍然不如通用 GPU**——这说明对于 IVF-PQ 这类大规模并行搜索，GPU 的通用计算能力已经足够高效，FPGA 的专用化收益不足以抵消 GPU 的架构优势。

### 3.4 SmartANNS（Samsung, USENIX ATC 2024 / ACM TOS 2025）

**全称**：SmartANNS: SmartSSD-based Acceleration of Approximate Nearest Neighbor Search

#### 加速定位

SmartANNS 在 Samsung SmartSSD（含 Xilinx Kintex UltraScale+ FPGA）上加速 ANN 搜索，利用 SSD 内的 FPGA 计算能力减少主机-设备数据搬运。

| 算法阶段 | 加速方式 | 详述 |
|---------|---------|------|
| **③ 候选筛选** | ✅ **分层索引** | **两层索引**：粗粒度层（快速定位簇）+ 细粒度层（精确候选筛选），减少 **80%** 的总距离计算量 |
| **④ 距离计算** | ✅ **FPGA 加速** | SmartSSD 上的 FPGA 完成候选向量的距离计算，无需经 PCIe 回传主机 |
| **① 索引构建** | ⚠️ 部分 | 索引结构在主机构建，但向量数据存储在 SmartSSD 上 |
| **② 图遍历** | ❌ | 非图结构，基于分层索引 |
| **⑤ Top-K 排序** | ⚠️ 主机完成 | FPGA 计算距离后，主机做最终排序 |

#### 核心创新：分层索引

```
主机侧          SmartSSD (FPGA) 侧
粗粒度索引         细粒度索引 + 向量数据
(选择候选簇)    →   (簇内精确距离计算)
   ↓                   ↓
 减少 80% 的距离计算    FPGA 流水线加速剩余 20%
```

#### 性能数据

| 指标 | 数值 | 加速阶段 |
|------|------|---------|
| QPS vs CSDANNS（前代） | **10.7×** | **③+④ 筛选+距离** |
| 距离计算减少 | **80%** | **③ 分层筛选** |
| 4 SmartSSD 扩展效率 | **94%**（3.76×/4×） | **③+④ 跨设备并行** |
| 数据集规模 | 4 个十亿级数据集 | — |

> **来源**：[ACM TOS 2025](https://dl.acm.org/doi/abs/10.1145/3736589)

**价值与代价**：
- ✅ **价值**：**"先算法优化再硬件加速"的顺序是对所有方案最有益的启示**——80% 距离减少来自分层索引剪枝，FPGA 加速剩余部分；在真实 Samsung SmartSSD 硬件上实测（非仿真），**10.7× QPS vs 前代 CSDANNS**；4 SmartSSD 扩展效率 **94%**（3.76×/4×），远超 GPU 多卡（60-80%）
- ⚠️ **代价**：**算法减少 80% 工作量后，FPGA 加速对系统总吞吐的边际贡献仅 ~1.6×**（80% 算法收益 + 20%×3× 硬件收益）；SmartSSD 为特定硬件产品，方案不可迁移至通用 SSD；分索引在数据更新时（>5%/次）重构代价较高——增量更新效率未公开

**深度洞察**：SmartANNS 的"80% 距离减少"是其最值得深挖的数字，但需要理解这个 80% 来自**分层索引的剪枝效果**而非硬件加速本身。这意味着 SmartANNS 的加速路径是：**先通过算法优化（分层索引）大幅减少工作量，再用 FPGA 加速剩余工作**。这个"先算法后硬件"的顺序很重要——80% 的收益来自算法层面，FPGA 提供的是剩余的 20% 的加速。**当算法已经减少了 80% 的计算量后，FPGA 对剩余 20% 的加速（即使只有 2-3×）对系统整体的贡献只有 ~1.5-2×（因为 20% × 2-3 = 40-60%，远不及 80% 的算法收益）。** 这揭示了一个通用原则：**在向量搜索领域，算法优化的杠杆率远高于硬件加速**。SmartANNS 的真正亮点是 4 SmartSSD 的 94% 扩展效率——这意味着跨 SSD 的分布式搜索可以接近线性扩展，这是 GPU 多卡扩展（通常 60-80%）难以达到的。

### FPGA 路线小结

| 方案 | FPGA | 核心加速阶段 | 功耗 | 性能亮点 | 规模 |
|------|------|------------|------|---------|------|
| **IBM VSM** | FCM 内嵌 FPGA | **③④⑤ 全流水线** | ≈25W | **>17K QPS**, 30× 能效 vs GPU | **6B** |
| **Falcon** | Alveo U250 | **②④ 图遍历+距离** | 55–62W | 4.3–19.5× 延迟 ↓ | 10M |
| **FANNS** | Alveo U55c | **③④⑤⑦ IVF-PQ 全流程** | ~75W | 20.79× 吞吐 vs 基线 FPGA | 1B |
| **SmartANNS** | Kintex US+ | **③④ 分层筛选+距离** | ~30W | 10.7× QPS, 80% 距离 ↓ | **1B** |

**核心启示**：
- FPGA 在 **小 batch / 单查询延迟** 上具有独特优势（Falcon sub-ms 级延迟，VSM 32 路流水线 K 无关排序），这是 GPU 在大批量高延迟优化的另一面
- **近存储 FPGA（VSM/SmartANNS）** 是 FPGA 路线中最有差异化价值的方向——将计算嵌入 SSD 内部，消除 PCIe 数据搬运，GPU 无法复制此架构
- **VSM 是 FPGA 路线中加速覆盖最完整的方案**——从 NAND 读取 → ECC 解码 → 32 路 MAC 距离计算 → 硬件 Max-Heap 排序，全流水线在 SSD 控制器内完成
- FPGA 相对于 ASIC 的 **可重构性** 在向量搜索场景中价值有限（算法已趋稳定），但在 SSD/存储硬件复用（FCM 双模切换）场景下有战略价值
- **FPGA 的规模受限**——Falcon 仅 10M（受片上 BRAM/URAM 限制），VSM 达 6B（存储内计算不依赖片上存储），说明近存储 FPGA 的可扩展性远优于纯 FPGA 加速卡方案
- **GPU batch 反超风险**——Falcon 对比 V100 在小 batch 出色，但大 batch 下 GPU 凭借 HBM 带宽反超（12× 带宽优势）；此差距在 H100/B200 上进一步扩大

## 七、深度洞察与批判性分析

> 本报告不是终点。真正的洞察力不在于"知道有什么方案"，而在于识别这些方案背后的**前提假设、方法论缺陷和未被言明的权衡**。以下尝试将分析提升一个层次。

### 7.1 证据质量分级——不是所有"× vs GPU"都可比

本报告 95 条声明中，通过验证的仅 17 条。这本身就揭示了一个深层问题：**该领域的性能数据普遍缺乏可比性和独立复现**。以下是证据质量的等级框架：

| 证据等级 | 含义 | 代表方案 | 常见陷阱 |
|---------|------|---------|---------|
| **A级：已流片+独立第三方复现** | 多团队在独立环境下验证 | ❌ **尚无** | — |
| **B级：已流片/物理实测+同行评审** | 真实芯片测量，但仅来自原团队 | ✅ Falcon, VSM, SmartANNS, IBM HERMES, Spyre | GPU 基线过时、工况选择性报告 |
| **C级：仿真/综合+同行评审** | 架构级或 cycle-level 模拟 | ⚠️ DReX, ANNA, NDSEARCH, Cosmos, **ANSMET** | 仿真假设偏离实际、热/功耗模型不完整 |
| **D级：厂商宣传/博客/预印本** | 未经验证的市场或技术声明 | ⚠️ SK Hynix "3×" 声明 | 选择性指标、缺失比较基准 |

**深度洞察：该领域最大的问题不是缺乏方案，而是缺乏可复现的公平对比。** 当 Falcon 的 GPU 基线是 V100（2017）、DReX 的 GGNN 基线可能未经充分调优、大多数方案在仿真而非真实硬件上运行——所谓的"×倍加速"需要被解读为"在特定假设下的上限估计"而非"实际可获得的加速"。

---

### 7.2 GPU 进化曲线——专用加速器的"时间窗口"在关闭

这是整个报告中最关键的单一洞察，也是各方案论文中最少讨论的维度。

**GPU 带宽演进 vs 专用加速器**：

| 发布时间 | GPU | HBM 带宽 | 带宽增益 | 对应专用加速器声称的优势 |
|---------|-----|---------|---------|----------------------|
| 2017 | V100 | ~900 GB/s | 基线 | Falcon: 26.9× 能效 |
| 2020 | A100 | ~2,000 GB/s | **2.2× vs V100** | NDSEARCH: 120.87× 能效 |
| 2022 | H100 | ~3,350 GB/s | **3.7× vs V100** | DReX: 6.7× vs H100 |
| 2024 | B200 | ~8,000 GB/s | **8.9× vs V100** | — |
| 2026 | Rubin | ~10,000+ GB/s | **11×+ vs V100** | — |

**深度洞察**：
- Falcon 对比 V100 的 26.9× 能效优势，在 B200（2024）面前可能已缩小至 **3-5×**
- 专用加速器论文通常基准测试的是发布时的 GPU——到论文被引用时，该 GPU 已落后 1-2 代
- 这意味着：**所有声称的加速比都有一个"半衰期"**。一篇 2022 年的专用加速器论文（如 ANNA），到 2026 年其对比优势已被 GPU 自然迭代侵蚀至少 50-70%
- **真正的考验不是"能否比当前 GPU 快"，而是"能否在 3 年后仍比当时的 GPU 快"**——对 ASIC 而言这是致命挑战，对 FPGA 而言稍好（可重配置部分补偿），对 GPU 自身则不存在此问题

---

### 7.3 成本归一化——芯片级数据远不等于 TCO

几乎所有方案都报告芯片级功耗和性能，但**系统级 TCO** 几乎从未被完整讨论。以下是一个归一化尝试：

| 成本维度 | GPU（H100 8卡） | FPGA（VSM 48路） | ASIC（Spyre 12卡） |
|---------|---------------|-----------------|-------------------|
| 硬件采购（1年摊薄） | ~$50,000（参考 AWS p5 1年预留） | ~$3,000-5,000（单 FCM SSD + FPGA） | ~$15,000-25,000（预估）|
| 基础设施/散热 | ~3,500W × $0.12/kWh = ~$3,680/年 | ~1,200W × $0.12/kWh = ~$1,260/年 | ~900W × $0.12/kWh = ~$945/年 |
| 软件/集成成本 | ~$20,000-50,000（Faiss/cuVS 生态，人员丰富） | ~$100,000-200,000（定制 FPGA 开发） | ~$200,000-500,000（ASIC 工具链）|
| **全周期 3 年 TCO** | **~$220,000-320,000** | **~$106,000-156,000** | **~$230,000-530,000** |

**深度洞察**：
- FPGA 看似硬件成本低，但 **3 年 TCO 受软件/集成成本支配**（定制 FPGA 开发人力昂贵）
- GPU 看似功耗高，但 **生态成熟度大幅压低软件成本**
- ASIC 只有在 **足够大规模出货**（>10 万片）时，NRE 摊薄后才有经济性——目前所有学术 ASIC 方案均未进入此阶段
- **"便宜的芯片" ≠ "低 TCO 的方案"**——这是论文最容易忽略的维度

---

### 7.4 三个核心矛盾——这个领域的根本性张力

| # | 矛盾 | 张力描述 | 影响 |
|---|------|---------|------|
| **矛盾一** | **可重构性 vs 效率** | FPGA 牺牲效率换取可重构；ASIC 牺牲灵活性换取极致能效；GPU 在两者之间但功耗高 | 不存在同时满足三者的方案——选择就是做取舍 |
| **矛盾二** | **创新速度 vs 验证深度** | 学术方案（ANNA/DReX/NDSEARCH）2-3 年提出新架构，但通常只到仿真；产业方案（VSM/Spyre）验证充分但迭代慢 | 该领域的大部分"创新"可能永远不会成为产品 |
| **矛盾三** | **学术追求 vs 产业可用性** | 学术论文追求 "×倍 vs GPU"（用旧 GPU 基线放大优势）；产业追求 "可用、可维护、可扩展" | 学术界报告的加速比在落地时往往缩水 50-80% |

**深度洞察——一个判断框架**：

```
评估任意加速方案时，问三个问题而不是一个：

❌ "它比 GPU 快多少倍？"（受基线选择操纵）
✅ "它比发布时的最新 GPU 快多少？"（控制时间因素）
✅ "这个优势在 3 年后还成立吗？"（评估持久性）
✅ "把它集成到现有系统，总成本是多少？"（评估可落地性）
```

### 7.5 对读者"人"的洞察——不同角色的关注点分歧

技术报告通常假设所有读者关注同样的问题。但向量检索硬件加速领域，不同角色的利益分歧显著：

| 角色 | 首要关注 | 次要关注 | 可能被误导的方向 |
|------|---------|---------|----------------|
| **算法研究员** | 能否在约束条件下获得最高 recall/最低延迟 | 方案可复现性、论文发表潜力 | 忽略实际系统集成成本和功耗限制 |
| **系统架构师** | TCO、运维复杂度、可扩展性 | 与现有基础设施的兼容性 | 被单个数字指标（如 TOPS/W）吸引，忽略整体 |
| **硬件工程师** | 架构创新点、工程可行性 | 量产路线图、工具链成熟度 | 低估 GPU 生态的快速迭代能力 |
| **技术决策者/VP** | 投资回报周期、技术风险、供应商锁定 | 人才可用性、替代方案 | 被厂商营销的 "×倍加速" 引导，缺乏交叉验证 |

**深度洞察**：该领域的"解决方案"只有在特定角色的视角下才是最优的。一个方案的"最优"在 VP 看来可能是最高风险的选择——**技术报告的核心价值不是给出"最佳方案"，而是让每个角色看清自己需要关注的那个维度**。
