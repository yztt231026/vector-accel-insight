## 二、数据精选（一页 PPT 放 3-4 个数据最有效，这里给出 TOP 5 候选）

### 候选 ①：规模断层——GPU 显存追不上数据增长

| 数据 | 来源 |
|------|------|
| **1B 个 768 维 FP16 向量 ≈ 3 TB** | 基础计算 |
| H100 80GB 单卡仅能容纳 **~21M 个 768D FP16 向量** | 基础计算 |
| 训练数据量每 **12-18 个月翻一番**，HBM 容量每代仅增 **~2.4×** | 行业共识 |
| 差距：数据增长（2-3×/年）> HBM 容量增长（~1.5×/年） | 推算 |

**备选句式**："今天你需要 19 张 H100 才能存下 1B 向量——这还没算索引的开销。"

### 候选 ②：RAG 延迟反转——搜索取代 LLM 成为瓶颈

| 数据 | 来源 |
|------|------|
| 在 **128M 个 2048 维向量**上，向量搜索占首 Token 延迟 **85.8%** | VectorLiteRAG, 2025 |
| CPU 搜索比 LLM 预填充阶段**慢 2×**——搜索不再是"免费查找" | VectorLiteRAG, 2025 |
| Agent 工作负载：人类每几分钟一次查询，Agent **每秒数百次** | Qdrant CEO, 2026 |

**备选句式**："你花了 90% 的预算在 GPU 上跑 LLM，但用户等的那一秒里有 85% 耗在向量搜索上。"

### 候选 ③：索引构建耗时——从分钟到天的量级跨越

| 数据 | 来源 |
|------|------|
| CPU HNSW 构图 10M 1024D：**18-22 分钟** | CAGRA, ICDE 2024 |
| GPU CAGRA 构图同规模：**45 秒**（**~30× 加速**） | CAGRA, ICDE 2024 |
| IBM CAS 100B 索引：CPU 需 **120 天** → GPU 仅 **4 天** | IBM Research, 2026 |

**备选句式**："构建一次索引从时分秒级变成天级——在生产环境中这意味着你无法及时上线新数据。"

### 候选 ④：TCO 剪刀差——GPU 成本 vs 收益

| 数据 | 来源 |
|------|------|
| 百度 1536D-5M 实测：48 CPU 实例 **$28,800/月** vs 6 GPU 实例 **$3,600/月**（**8× 降本**，4.9× QPS 提升）| 百度公开基准 |
| GPU 硬件占自托管向量数据库预算的 **80%+** | 行业报告 |
| GPU 的成本降低 + 性能提升**不一定**带来 TCO 最优——混合方案（GPU 构图+CPU 查询）才是产业共识 | 本报告综合分析 |

**备选句式**："GPU 构图+CPU 查询正在成为产业共识——因为纯 GPU 搜索在大规模场景下不经济。"

### 候选 ⑤：内存墙——向量搜索独特的计算特征

| 数据 | 来源 |
|------|------|
| 向量搜索（L2/IP）算术强度：**0.5-0.75 FLOPs/byte**（内存受限）| 本报告分析 |
| CNN 推理：10-50 FLOPs/byte（计算受限）| 行业共识 |
| LLM Prefill：50-200+ FLOPs/byte（计算受限）| 行业共识 |
| **Tensor Core 数千 TFLOPS 对向量搜索几乎无用**——瓶颈在带宽 | 本报告核心洞察 |

**备选句式**："向量搜索的算术强度只有 0.75 FLOPs/byte——这意味着你花的 GPU 钱里，大部分买的是你不需要的东西。"

## 四、页面文案模板

### 标题区（建议 15 字以内）

> **主标题**：向量检索——AI 管线中被低估的瓶颈
> **副标题**：当数据规模突破硬件上限，加速不再是选择而是必然

### 正文区

**左侧：行业正在经历三重重压**

```
① 规模断层    1B 向量 ≈ 3 TB，H100 能装 → 19 张
     ↓
② 延迟反转    搜索占首 Token 延迟 85.8%（128M 向量级）
     ↓
③ 成本压力    GPU 占自托管预算 80%+，混合方案才是 TCO 最优解
```

**右侧：核心矛盾——向量搜索是内存密集型，不是计算密集型**

```
算术强度对比（FLOPs/byte）
                        ↓
  Vector Search  0.75  ← 🔴 内存带宽瓶颈
  CNN Inference  10-50 ← 🟢 计算瓶颈
  LLM Prefill   50-200 ← 🟢 计算瓶颈
                        ↓
「Tensor Core 的数千 TFLOPS 对向量搜索几乎无用」
```

**底部结论条**：

```
所有高效的硬件方案——GPU（近存构图）、ASIC（存内计算）、FPGA（SSD 内流水线）—— 
都在做同一件事：缩短数据搬运路径。
```

## 六、讲稿脚本（90 秒版本）

```
（前 30 秒——痛点）
"向量检索正在成为 AI 管线中被低估的瓶颈。三组数据：
第一，规模断层——1B 向量 3TB，你需要 19 张 H100 才存得下；
第二，RAG 延迟反转——在百万级向量上，搜索占首 Token 延迟的 85.8%；
第三，成本压力——GPU 占了自托管预算的 80% 以上，但混合方案告诉我们需要 GPU 的不一定是搜索环节。

（中间 30 秒——核心矛盾）
这三组数据的共同根源是什么？看这里——向量搜索的算术强度只有 0.75 FLOPs/byte。
CNN 推理是 10-50，LLM 是 50-200。
这意味着向量搜索的瓶颈不在计算，在内存带宽。
Tensor Core 的数千 TFLOPS——它需要被喂饱数据才能发挥作用——而向量搜索正好喂不饱它。

（最后 30 秒——定调）
所以所有高效的硬件方案——无论是 GPU 构图、ASIC 的存内计算、还是 FPGA 的 SSD 内流水线——
它们都在做同一件事：把计算往数据那里推，而不是把数据往计算这里拉。
这彻底颠覆了传统的加速思路。
后面三页，我们逐一检视这三个方向。
```
