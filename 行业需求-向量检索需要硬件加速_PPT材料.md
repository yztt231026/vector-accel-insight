## 二、角度一：算法视角——向量检索算法的哪些阶段需要硬件加速？

### 2.1 向量检索的 7 个阶段与瓶颈分析

| 阶段 | 计算特征 | 内存访问特征 | 占搜索总时间 | CPU 是否擅长？ |
|------|---------|------------|------------|--------------|
| **① 索引构建** | 计算密集型，大量距离计算 | 顺序+随机混合 | 离线阶段（分钟~天） | ❌ 越大规模越不可接受 |
| **② 图遍历/索引导航** | 逻辑密集，距离计算+比较 | **随机访问主导** | 5-15% | ⚠️ 随机访存导致 cache miss 率 >80% |
| **③ 候选点筛选** | **带宽密集型** | **顺序扫描主导** | 60-80% | ❌ 顺序带宽受限于内存带宽 |
| **④ 距离计算** | **核心瓶颈，向量化友好** | **带宽密集型** | 包含在③中 | ❌ 受限于 SIMD 宽度和带宽 |
| **⑤ Top-K 排序** | 计算/逻辑混合 | 寄存器/片上 SRAM | 5-10% | ✅ 较大 K 时 CPU 的堆排序高效 |
| **⑥ 重排序/精确计算** | 计算密集型 | 取决于候选集大小 | 0-10%（可选） | ⚠️ 可用，但候选集大时慢 |
| **⑦ 向量量化/解码** | 查表+加法 | 码本在片上 | 包含在③④中 | ⚠️ 受限于查表并行度 |

> **核心发现：阶段 ③④②——候选筛选、距离计算、图遍历——占据搜索总时间的 70-90%，且它们的共同瓶颈是"内存带宽"而非"计算吞吐"**

### 2.2 为什么 CPU 在向量搜索上天然低效？

**瓶颈一：顺序扫描带宽受限**

```
CPU DDR5-4800 理论带宽：~76.8 GB/s  ← 共享给所有核心
GPU HBM3 理论带宽：~3,350 GB/s      ← 单卡就有 44× 差距

一台双路服务器 CPU 带宽 < 1 张 H100 GPU 的一半
但 GPU 的 HBM 也远远不够存十亿级向量
→ 问题不在"谁带宽更高"，在"带宽永远不够用"
```

**瓶颈二：随机访问的 cache miss**

```
CPU HNSW 图遍历中，cache miss 率 >80%
——每次图遍历都要从 DDR 中 fetch 邻居节点
——CPU 的 L1/L2/L3 cache 对图搜索几乎无用
——GPU 的 shared memory 同样面临此问题
→ 图遍历的随机访问模式是架构层面的不匹配
```

**瓶颈三：SIMD 利用率不足**

```
CPU 的 AVX-512 可以一次处理 16 个 FP32 距离计算
——但前提是数据在连续内存中
——向量存储是行优先，距离计算是列间的
——数据整理（gather）开销抵消了 SIMD 收益
→ 这也是为什么 FAISS GPU 的 Interleaved-by-32 能带来 2.7× 提升
```

### 2.3 算法视角结论

```
向量检索算法的核心瓶颈不在"计算太慢"，而在"数据搬不动"：
  - 距离计算需要大量带宽（算术强度 0.75 FLOPs/byte）
  - 图遍历的随机访问让 cache 失效
  - SIMD 的 gather 开销抵消了向量化收益

CPU 的架构设计（大缓存 + 通用计算单元）是为控制流密集而非数据流密集任务设计的
→ 向量搜索需要的是"靠近数据的小算力"而非"远离数据的大算力"
→ 这正是硬件加速（近存计算、存内处理、FPGA 流水线）能发挥作用的地方
```

## 三、角度二：规模视角——向量数据库的规模增长如何倒逼硬件加速？

### 3.1 每个规模量级都在突破架构上限

> 从 10M 到 100B 亿，每跨越一个数量级，原有的架构假设就失效一次。

```
                                 架构失效              必需硬件
10M ─────────────────────  单机 CPU 可承载 ───── 无需专用硬件
                          ↓ 
100M ────────────────────  CPU 构图小时级 ───── GPU 构图首次进入
                          ↓                         决策
1B ──────────────────────  GPU VRAM 溢出 ────── FPGA 近存开始
                          ↓                        差异化
10B ─────────────────────  PCIe 搬运成为主瓶颈 ── 存算融合成为必须
                          ↓
100B+ ──────────────────  单节点不可行 ──────── 多级架构协同
```

### 3.2 每个量级的具体痛点

| 规模 | 总数据量 | 算法瓶颈 | 硬件瓶颈 | 后果 |
|------|---------|---------|---------|------|
| **~10M** | ~15 GB | CPU HNSW 可承受 | 无 | 纯 CPU 足够 |
| **~100M** | ~150 GB | CPU 构图小时级；IVF-PQ 量化成为必须 | 单 GPU VRAM 溢出（H100 80GB） | GPU 构图首次成为"更好选项" |
| **~1B** | ~1.5 TB | CPU 构图天级不可接受；ANN 精度损失 <90% | 必须量化（PQ/SQ），GPU VRAM 完全不夠 | GPU 构图从"更好"变"必须"；FPGA 近存开始展示价值 |
| **~10B** | ~15 TB | 分层索引+量化+分布式成标配 | 数据搬运开销主导；PCIe 带宽成为瓶颈 | GPU 退出在线路径；近存储/存内计算进入主力 |
| **~100B+** | ~150 TB | 存储计算融合；精确/ANN 双轨 | 能耗墙；系统复杂度爆炸 | FPGA（VSM 48×）+ ASIC（Spyre 精确）多级协同 |

### 3.3 三层瓶颈的逐级引爆

> 不是所有痛点同时出现——每增长一个量级，引爆一层新瓶颈。

```
量级             1K  10K  100K  1M  10M  100M  1B  10B  100B
                    │    │     │   │    │    │   │    │
① 内存墙    ◄──────┴────┴─────┴───┴────┴────┴───┴────┴── 始终存在
② VRAM限    ◄─────────────────────┴────┴────┴───┴────┴── ≥100M 起
③ PCIE搬    ◄────────────────────────────────┴───┴────┴── ≥1B 起
④ 能耗墙    ◄────────────────────────────────────────┴── ≥10B 起
```

### 3.4 规模视角结论

```
不是"硬件加速突然变重要了"——是数据规模一直在涨，但过去 3 年从 100M 跃迁到了 1B+。
在 100M 级，CPU 够用，GPU 是"更好"；
在 1B 级，CPU 不够用，GPU 是"必须"；
在 10B 级，GPU 也不够用了。

→ 硬件加速的出现时机不是巧合，是被规模逼出来的。
```

## 四、两面夹击——算法 × 规模 的合力

### 4.1 两条线的汇合

```
算法视角：向量检索是 bandwidth-bound + 随机访问密集
              ↓
         CPU 架构天然不适合
              ↓
         需要"数据附近的小算力"
              ↓
         ┌─────────────────────┐
         │     两面夹击         │
         │  硬件加速成为必然      │
         └─────────────────────┘
              ↑
规模视角：10M→100M→1B→10B 每级突破上限
              ↑
         CPU→GPU→FPGA/ASIC 逐级失效
              ↑
         不是选择问题，是生存问题
```

### 4.2 三个方向回应不同的"算法-规模"组合

```
算法瓶颈 ↓     规模 →  100M 级          1B 级             10B 级以上
──────────────────────────────────────────────────────────────────
③ 候选筛选          GPU IVF-PQ      FPGA SmartANNS      存算融合
                     （带宽并行）     （80% 距离减少）    （CAS 存储层搜索）
④ 距离计算          GPU Tensor Core  FPGA VSM 32 路      ASIC HERMES
                     （矩阵乘映射）    （近存流水线）      （模拟存内计算）
② 图遍历            CAGRA 构图       NDSEARCH LUN NDP   Falcon DST
                     （GPU 并行）     （NAND 内计算）     （延迟同步遍历）
```

### 4.3 一个量化的横截面——1B 级搜索的典型成本分布

> 在十亿级部署中，功耗和延迟的分布揭示了同样的结论：**数据搬运是主导开销**。

```
一次 1B 级向量搜索的成本分布（归一化）：
                            计算  9%   ← 实际距离计算
数据传输 ██████████████████████ 62%  ← PCIe + DRAM 搬运
        等待 ████████ 22%           ← 内存访问延迟
            排序 7%                ← Top-K
```

## 六、讲稿脚本（90 秒版本）

```
第一部分：算法视角（30 秒）

"向量检索不是一个大一统的计算任务——它可以拆成 7 个阶段。
其中 70-90% 的时间都花在候选筛选、距离计算和图遍历上。
这三个阶段的共同特点是什么？——内存带宽密集型。
算术强度只有 0.75 FLOPs/byte，而 CNN 推理是 50，LLM 是 200。
CPU 的架构设计——大缓存、分支预测、SIMD——是为控制流和规整计算设计的。
向量搜索的数据流特征正好和 CPU 的强项错配。
所以你需要的是靠近数据的处理单元，而不是更多远离数据的算力。

（过渡）但光是算法特征还不够——还要看规模。

第二部分：规模视角（30 秒）

从 10M 到 100B 亿，每跨越一个数量级，原有的架构就失效一次。
100M 级：CPU 构图开始慢，GPU 首次有用；
1B 级：单 GPU VRAM 装不下了，必须量化，必须近存；
10B 级：PCIe 搬运占据了 60% 以上的功耗和延迟。
这三年，主流部署从 100M 跃迁到了 1B。
所以硬件加速不是突然变重要的，是被规模逼出来的。

第三部分：定调（30 秒）

算法特性和规模增长，从两个方向同时挤压——这是一个"两面夹击"的局面。
后面三页我来拆解三个方向。
每个方向我都先问：它解决的是哪个阶段的瓶颈？它适合哪个规模量级？
我们不谈厂商，我们谈你的痛点在哪。
```
