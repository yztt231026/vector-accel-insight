### 问题 ①：距离计算——带宽瓶颈

**对应行业需求位置**：第 2.2-2.3 节，算法视角的核心矛盾

**一句话痛点**：
> "距离计算占搜索总时间的 60-80%，但算术强度仅 0.75 FLOPs/byte——瓶颈不在计算，在带宽"

**目标听众**：架构师、算法工程师——理解"为什么 GPU 的 TFLOPS 帮不了距离计算"

#### Slide 1：Falcon + DST（ETH Zurich, VLDB 2025）—— FPGA 图遍历加速

| 要素 | 内容 |
|------|------|
| **成果类型** | 🎓 学术界（FPGA 物理实测） |
| **针对问题** | 图遍历+距离计算（②+④），带宽瓶颈导致 CPU/GPU 低效 |
| **核心创新** | Directed Sonar Traversal（DST）——延迟同步减少片上-片外数据交换 |
| **关键指标** | CPU（Xeon）对比：**4.3× 延迟降低** → 回答"对比你能获得的基线" |
| | GPU（V100）对比：**19.5× 延迟降低**，**26.9× 能效** |
| | 功耗：55-62W（vs GPU 183-324W）|
| | DST vs BFS：**1.7-2.9×** 额外加速，R@10 提升 0.14-4.93% |
| **规模上限** | ~10M（FPGA 片上 BRAM 限制）|
| **局限** | GPU 基线 V100（2017），到 H100/B200 优势缩至 3-7× |
| **国产化视角** | |
| ✅ **价值** | DST 算法本身是通用算法贡献，不绑定 FPGA 平台；延迟同步思想可移植到任何处理器架构——国产 CPU（飞腾/鲲鹏）或国产 NPU（昇腾）同样可以受益 |
| 💡 **建议** | 优先将 DST 算法作为软件优化在国产 CPU（飞腾 S2500、鲲鹏 920）上验证效果（无需 FPGA）；若算法收益 >1.5× 则考虑映射到国产 FPGA（中科亿海微、安路科技、紫光同创）；Alveo U250 受 EAR 管制，但 DST 算法 + 中低端 FPGA 组合可能覆盖 70% 的实用场景 |

**讲稿脚本（60 秒）**：
```
距离计算是向量搜索的核心瓶颈——占 60-80% 的时间。

ETH Zurich 的 Falcon 用 FPGA 做了三件事：
第一，DST 算法把图遍历中的同步次数减少了 1.7-2.9×；
第二，全流水线在 FPGA 片上完成，功耗只有 55-62W；
第三，相比 V100 GPU，做到了 26.9× 的能效提升。

但注意：GPU 基线是 V100——到今天的 H100，带宽涨了 3.7×，
这个 26.9× 的优势已经缩小到了大概 3-7×。
Falcon 证明了方向正确，但 FPGA 的 10M 规模上限意味着：
最需要加速的十亿级场景，它用不上。

国产化视角：DST 算法本身不依赖特定硬件，Alveo U250 受管制，
但 DST 的延迟同步思想可以在国产 FPGA 或 CPU 上落地。
```

#### Slide 2：QTT Tensor Core（arXiv 2026）—— 把 L2 距离变成矩阵乘

| 要素 | 内容 |
|------|------|
| **成果类型** | 🎓 学术界/预印本（GPU 仿真，未同行评审）|
| **针对问题** | 距离计算（④）——证明 Tensor Core 在向量搜索中有用 |
| **核心创新** | QTT 将向量分解为张量序列，距离计算 = 一次 GEMV（Tensor Core）|
| **关键指标** | **1B 向量 / 单 H100 / 38ms / 100% recall** （serving size 66GB）|
| | 500M 384D FP16：3.1ms p50，98% recall |
| | 反常现象：recall 随规模递增（400M→96%，1B→100%）|
| **精度** | FP16/BF16/FP8/INT8/INT4 |
| **局限** | arXiv 预印本，未同行评审；仅 384D 验证；高维（768D+）预估 80-150ms |
| **国产化视角** | |
| ✅ **价值** | L2→矩阵乘映射是一个数学等价变换，不依赖 NVIDIA 硬件——华为昇腾 910B 的 Cube Unit（达芬奇架构）同样支持 FP16/BF16 矩阵乘，理论可执行相同映射；这一思路可能在国产算力上**复现部分优势** |
| 💡 **建议** | 与华为昇腾团队沟通 CUDA→CANN 迁移可行性——QTT 的核心是一组 GEMV，CANN 的矩阵乘 API 可以直接承接；在昇腾 910B 上复现 38ms@1B 的可行性评估是关键的第一步；若验证成功，可构建完全自主可控的高精度向量搜索方案（昇腾 + 向量数据库） |

**讲稿脚本（60 秒）**：
```
如果 Tensor Core 不只是做矩阵乘，还能算距离呢？
这个想法把 L2 距离拆成 ||x||² + ||y||² - 2⟨x,y⟩，
内积部分正好映射到 Tensor Core 的 MMA 指令。

QTT 更进一步——把向量分解成张量序列，
一次查询只需要一次 GEMV。
结果是：单张 H100，1B 向量，38 毫秒，100% recall。
注意这个反常现象：recall 随数据量增大反而提升——
400M 时 96%，1B 时 100%。

但请谨慎：这目前只是 arXiv 预印本。
如果被复现、被扩展到更高维数据，
它会从根本上挑战"近似搜索在大规模下必然牺牲 recall"这个领域假设。

国产化视角：L2→矩阵乘是数学变换，不绑定 NVIDIA。
昇腾 910B 的 Cube Unit 同样能跑矩阵乘。
QTT 的关键不是 GPU 型号，而是张量分解的思想。
```

#### Slide 3：IBM HERMES（Nature Electronics 2023）—— 极低功耗的距离计算

| 要素 | 内容 |
|------|------|
| **成果类型** | 🏢 产业界（IBM Research，✅ 14nm 已流片实测）|
| **针对问题** | 距离计算（④）——能耗墙下的极致能效 |
| **核心创新** | PCM 模拟存内计算——MVM 在 crossbar 阵列中以模拟方式完成 |
| **关键指标** | **6.5W / 63.1 TOPS（9.76 TOPS/W）** — H100 的 ~14× 能效 |
| | 核分类精度差 <1% vs FP32；LRA 注意力差 0.08pp |
| **工艺** | 14nm CMOS 混合信号，64 核 × 256×256 PCM crossbar |
| **局限** | **~3-4 bit 有效精度**；研究芯片非产品；PCM 写耐久性~10^8 次 |
| **国产化视角** | |
| ✅ **价值** | 模拟存内计算不受先进数字工艺（如 EUV）出口管制限制——国内 28nm/22nm 成熟工艺也可实现有竞争力的模拟 crossbar 设计；中科院微电子所、清华大学已有相变存储器（PCM）和阻变存储器（RRAM）研究积累 |
| 💡 **建议** | 关注北京大学、中科院微电子所、华中科技大学在存算一体芯片方面的进展；模拟存内计算在向量搜索粗筛阶段（低精度高吞吐）的实用化窗口可能在未来 2-3 年；与国内存储器厂商（长江存储/长鑫存储）探讨 PCM 工艺路径 |

**讲稿脚本（60 秒）**：
```
当功耗是红线，ASIC 就是答案。

IBM 的 HERMES 芯片——14nm 已流片，不是仿真——
64 个 PCM crossbar 阵列在模拟域完成矩阵-向量乘法。
6.5W 跑了 63.1 TOPS——作为对比，一张 H100 是 700W。
能效是 H100 的约 14 倍。

代价是精度：~3-4 bit 有效精度。
如果你要匹配 FP32 的召回率，需要额外做数字精修，
系统级功耗增加 2-3 倍，能效优势从 14× 降到约 5×。

所以 HERMES 最适合的不是高精度向量搜索，
而是低精度高吞吐的粗筛或嵌入生成阶段。

国产化视角：模拟存内计算不依赖先进数字工艺——28nm 足够。
国内在 PCM/RRAM 存算一体方向有深厚积累，
这条路在制裁环境下反而是相对可控的。
```

#### Slide 4：SmartANNS（Samsung, USENIX ATC 2024 / ACM TOS 2025）—— 减少 80% 距离计算

| 要素 | 内容 |
|------|------|
| **成果类型** | 🏢 产业界（Samsung，✅ SmartSSD 实物验证） |
| **针对问题** | 候选筛选（③）——用分层索引剪枝 80% 的候选向量 |
| **核心创新** | 两层索引：主机构建粗粒度索引，SmartSSD FPGA 完成细粒度簇内搜索 |
| **关键指标** | **10.7× QPS** vs 前代 SmartSSD 方案（CSDANNS）|
| | **80% 距离计算减少**（来自算法剪枝，非硬件加速）|
| | **4 SmartSSD 扩展效率 94%**（3.76×/4×）|
| | 硬件：Samsung SmartSSD（Xilinx Kintex UltraScale+ FPGA）|
| **局限** | 80% 算法收益 + 仅 ~1.6× 硬件收益；不可迁移到通用 SSD |
| **国产化视角** | |
| ✅ **价值** | **"先算法后硬件"的顺序是最有价值的部分**——80% 收益来自算法优化，不依赖特定硬件；Samsung SmartSSD（Xilinx FPGA）受管制，但算法层面的分层剪枝设计可直接用于国产 CPU/加速器 |
| 💡 **建议** | 将两层索引算法复用在国产 CPU+国产 SSD 架构上验证（飞腾+致钛/忆联），先确认算法收益是否 >70%；如有硬件加速需求再考虑国产 FPGA（中科亿海微、紫光同创）；核心原则：**算法红利不依赖禁运硬件** |

**讲稿脚本（50 秒）**：
```
Samsung 的 SmartANNS 做了一个聪明的选择：
先在算法层面减少工作量，再用硬件加速剩余部分。

两层索引粗筛 → 80% 的候选向量不需要被精确计算——
这 80% 是算法收益，不花任何硬件成本。
剩余 20% 由 SmartSSD 里的 FPGA 加速。

代价是当算法红利 80% 已实现时，
FPGA 对剩余 20% 的加速对系统总吞吐的边际贡献只有 ~1.6×。
核心启示：算法优化的杠杆率远高于硬件加速——
这条原则适用于所有三条路线。

国产化视角：80% 的算法红利不依赖三星/Xilinx 硬件。
在飞腾 CPU + 国产 SSD 上即可验证。
这是封锁环境下"优先榨取算法收益再考虑硬件"的范本。
```

#### Slide 5：ANSMET（清华大学, ISCA 2025）—— 混合早停，不做不必要计算

| 要素 | 内容 |
|------|------|
| **成果类型** | 🎓 学术界（清华，cycle-accurate 仿真）|
| **针对问题** | 候选筛选（③）——89% DRAM 获取无效，用早停避免 |
| **核心创新** | 混合粒度早停：部分维度+部分位（保守下界估计）→ 不满足阈值则提前终止 |
| **关键指标** | **~8× 加速 vs CPU**（NDP-Base 5.26× + ET 额外 1.52×）|
| | 每 NDP 单元仅 **0.06mm²**（22nm），可集成到 DDR5 DIMM 缓冲芯片 |
| | **无精度损失**——保守下界保证不误淘汰 |
| | 获取利用率：CPU 6.0% → ANSMET 11.1%（仍有 89% 无效）|
| **局限** | 仿真验证；仅 HNSW 验证；对动态数据集不友好 |
| **国产化视角** | |
| ✅ **价值** | **国产大学成果，完全自主可控**——论文来自清华大学，代码已开源（GitHub）；DIMM 级方案利用通用 DDR5 接口，不依赖禁运硬件（不需要 HBM、不需要先进 FPGA）；0.06mm² @22nm 的面积意味着国内代工厂（SMIC 22nm）可以流片 |
| 💡 **建议** | 这是当前环境下**最直接可落地的方案**之一：与清华大学合作，在 SMIC 22nm 流片验证 NDP 单元；DIMM 缓冲芯片的改造成本远低于修改 NAND 设计或定制 ASIC；优先在骨干网络 HNSW 上验证，再扩展到 IVF 等更多索引类型 |

**讲稿脚本（60 秒）**：
```
ANSMET 揭示了一个被整个领域回避的数字。

在 CPU 的 HNSW 搜索中，只有 6% 的 DRAM 数据是"有用"的——
其余 94% 被读取后经距离计算发现比当前的最优解还差，被丢弃。

ANSMET 的做法是：在获取过程中渐进式评估。
先读部分维度 → 计算下界 → 如果已不可能成为 Top-K → 立即停止。
再读部分比特位（MSB 优先）→ 进一步细化估计 → 仍不够好 → 停止。
最后才完整读取和精确计算。

最终获取利用率从 6% 提升到 11.1%——有改善，但 89% 仍然无效。
这个数字告诉我们：根本问题不是"获取效率不够高"，
而是"向量搜索的随机访问模式天生会浪费大量带宽"。

国产化视角：这是清华的成果，代码开源，
DDR5 DIMM 方案不受任何出口管制。
0.06mm² @22nm 可以用 SMIC 流片。
在封锁环境下，这是最有合作价值的方案之一。
```

#### Slide 6：CAGRA（NVIDIA, ICDE 2024）—— GPU 原生图索引 + 构图加速

| 要素 | 内容 |
|------|------|
| **成果类型** | 🏢 产业界（NVIDIA，✅ GPU 部署验证） |
| **针对问题** | 索引构建（①）+ 图遍历（②）——GPU 替代 CPU 构图，**30× 加速** |
| **核心创新** | 固定出度+单层图结构（牺牲算法优雅换取 GPU 执行效率）|
| **关键指标** | 构图：**45s vs CPU HNSW 22min**（10M 1024D，~30×）|
| | 搜索：batch=100 时 **49× QPS vs CPU** |
| | 单查询（batch=1）：3-9×（大幅缩水）|
| | 微信部署验证成本降低 **~70%** |
| **局限** | 全索引驻留 GPU VRAM（单卡~21M 768D FP16）；构图显存开销高 |
| **国产化视角** | |
| ✅ **价值** | 固定出度+单层图结构是**算法层面的创新**，可移植到昇腾 910B：A国 CUDA 核心的构图核（NN-Descent、IVF-PQ 初始化）可用昇腾 CANN 重构；单查询 3-9× 的加速比在国产 GPU 上可能复现 |
| 💡 **建议** | 与华为昇腾团队合作，将 CAGRA 的构图算法迁移至 CANN 编程模型；关注昇腾 910B 的显存容量（HBM2e 32GB/64GB）→ 单卡约能容纳 **~10M 768D FP16 向量**，适合中等规模场景；华为 MindSpore 已有 FAISS 兼容接口，迁移成本可评估；**构图加速是 GPU 最不可替代的环节，也是最值得优先迁移的** |

**讲稿脚本（60 秒）**：
```
HNSW 在 CPU 上很好——10M 级 22 分钟构图，毫秒级搜索。
但 100M 级时构图开始以小时计，1B 级以天计。
GPU 来解决这个问题。

CAGRA 用固定出度+单层图结构——牺牲了 HNSW 的算法优雅，
但换来了 GPU 的执行效率。构图 45s vs 22min——30× 加速。
微信用这个方案，基础设施成本降了 70%。

但 CAGRA 有三条"使用规则"必须遵守：
第一，全索引必须驻留 GPU VRAM——单卡上限大约 21M 向量；
第二，大批量（batch≥100）才有 49×，单查询仅 3-9×；
第三，构图过程显存开销是索引大小的 1.5-2 倍。

它解决的是"构图太慢"这个痛点。
但"大批量搜索延迟高"——这个它只解决了一半。
"小批量搜索延迟高"——它解决得更少。

国产化视角：固定出度图结构 + NN-Descent 构图算法本身不绑定 NVIDIA。
昇腾 910B 的 CANN 可以承接构图核心的 CUDA kernel 重构。
构图是 GPU 最不可替代的价值所在，也是最值得优先迁移到国产算力的环节。
```

#### Slide 7：NDSEARCH（Duke/Samsung, ISCA 2024）—— NAND 内图遍历加速

| 要素 | 内容 |
|------|------|
| **成果类型** | 🎓 学术界（Duke/Samsung 联合，cycle-accurate 仿真）|
| **针对问题** | 图遍历（②）+ 距离计算（④）——**在 NAND 闪存内部完成遍历** |
| **核心创新** | LUN 级近数据处理——比 chip-level（DeepStore）更细粒度的并行 |
| **关键指标** | 吞吐：vs CPU **31.7×**，vs GPU **14.6×**，vs SmartSSD **7.4×** |
| | 能效：vs CPU **178.68×**，vs GPU **120.87×**，vs SmartSSD **30.06×** |
| | 系统功耗仅 **26.32W**（PCIe 55W 预算内）|
| **局限** | **仿真而非实测，到真实芯片预估能效衰减至 30-60×**；LUN 级集成需修改 NAND 设计，工程投入 $50-100M+ |
| **国产化视角** | |
| ✅ **价值** | 概念验证了"NAND 内近数据处理对向量搜索的巨大潜力"——国产生态中，**长江存储（YMTC）** 是潜在合作伙伴；YMTC 的 Xtacking 架构在 3D NAND 工艺上有独立技术路线，理论上可在存储阵列和外围电路间集成计算逻辑 |
| 💡 **建议** | 与长江存储建立联合研究项目，评估在 YMTC Xtacking 架构中集成 NDP 的可行性；优先关注**chip-level NDP**（比 LUN 级更易落地，工程成本低一个数量级）；NDSEARCH 论文显示 chip→LUN 的额外收益仅 2.9×——chip-level 方案可以在 10-20% 的 NAND 设计改动下获得 60-70% 的收益 |

**讲稿脚本（50 秒）**：
```
图遍历为什么 cache miss？因为邻居节点在物理上不连续分布。
NDSEARCH 的回答是：不要试图把数据按连续的方式组织——

LUN 级——SSD 内部最细粒度的操作单元——每个单元内部完成图遍历和距离计算。
论文报告的能效比是 178× vs CPU——全文最高。

预估真实芯片可能在 30-60×——仍然很好，但不是 178×。
而且 LUN 级集成意味着三星或铠侠需要修改 NAND 晶圆设计，
工程投入数千万美元，量产周期 3-5 年。

国产化视角：长江存储的 Xtacking 架构提供了独立的技术路径。
```

**讲稿脚本（60 秒）**：
```
这是整场报告最有意思的设计——双模 SSD。
同一个硬件，平时是 FlashCore Module 块存储，
需要时重配置成 Vector Search Module——向量搜索加速器。

它解决的是数据中心硬件利用率低的问题——
SSD 通常只有 30-50% 利用率，GPU 昂贵且供给有限。
VSM 允许在存储硬件空闲时把它"借"来搜索。
不增加硬件采购，释放算力储备。

K=1 和 K=100 延迟相同——软件做不到这一点。

但条件：需要 10k+ 并发查询才能饱和。
企业 RAG 场景通常只有 100-1000 QPS——
低吞吐时 VSM 利用率仅 1-10%，能效优势无法兑现。

国产化视角：在硬件受限的环境下，"一物多用"的战略价值更大。
国产 SSD 控制器厂商（得一微、忆联）可以考虑在 SoC 中集成轻量搜索，
先从 8 路流水线开始，验证方向后再扩展。
```

## 三、国产化视角·全局建议

### 3.1 三条路径的国产化可行性排序

| 优先级 | 方向 | 代表方案 | 风险等级 | 预估落地周期 |
|--------|------|---------|---------|------------|
| 🔴 **立即启动** | **算法层优化（不依赖硬件）** | SmartANNS 的分层剪枝、CAGRA 的固定出度图、DST 延迟同步 | 低（纯软件） | 3-6 个月 |
| 🟡 **1-2 年** | **可获取硬件上的加速** | ANSMET DIMM NDP（SMIC 22nm）、昇腾上的 CAGRA 构图迁移 | 中 | 12-18 个月 |
| 🟢 **3-5 年** | **定制/合作硬件** | 长江存储 NAND+NDP、国产 SSD 控制器+搜索功能、存算一体 ASIC | 高 | 24-60 个月 |

### 3.2 技术封锁下的决策原则

```
原则①：算法优先于硬件
   → SmartANNS 证明了 80% 的收益来自算法——在封锁环境下，先榨干算法层面的优化空间

原则②：通用接口优先于定制硬件
   → ANSMET 的 DDR5 DIMM 方案比 NDSEARCH 的 NAND 内方案更易落地
   → 因为 DDR5 是标准接口，NAND 内集成需要修改晶圆设计

原则③：自主生态优先于兼容禁运生态
   → 昇腾 CANN + 飞腾 CPU + 国产 FPGA 的组合虽然单点性能弱于 H100，
   → 但整体系统可获取、可持续、可改进

原则④：一物多用优先于专用硬件
   → VSM 的双模思想在国产环境比在高性能环境更有战略价值
   → 因为硬件采购受限时，"复用"是比"最优"更重要的指标
```
