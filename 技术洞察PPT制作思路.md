# 向量检索硬件加速技术洞察 · PPT 制作思路

> **目标受众**：技术决策者 / 架构师 / 研发团队（具备向量检索基础认知，关注硬件选型和趋势判断）
> **建议时长**：30-45 分钟（~20 页）
> **风格基调**：客观、批判性、数据驱动，不给出"最佳方案"而是给出"判断框架"

---

## 一、叙事主线——PPT 的灵魂

这份 PPT 不是"三个技术路线的百科全书式对比"。它的主线是一个**层层递进的洞察链**，每一步都在推翻听众的默认假设：

```
┌─────────────────────────────────────────────────────────────────────┐
│  主线：向量搜索加速的核心不是"堆算力"，而是"消除数据搬运"——      │
│        但每个方案消除搬运的位置不同，这决定了它的优势和宿命        │
└─────────────────────────────────────────────────────────────────────┘
          ↓
[第一幕] 为什么需要加速？→ 因为市场爆炸、数据太大规模超过了GPU显存
          ↓ "所以堆更多 GPU 就好？"
[第二幕] 为什么堆GPU不是答案？→ 因为向量搜索是内存密集型，不是计算密集型
          ↓ "那看看其他方案？"
[第三幕] 三个方案都在消除数据搬运，但在不同层级：
          - GPU：把数据搬到HBM（但VRAM不够大）
          - ASIC：把计算塞进存储器（但精度受限、流片太慢）
          - FPGA：把计算放到SSD内部（但批量吞吐不如GPU）
          ↓ "那到底该选哪个？"
[第四幕] 这取决于你的瓶颈在哪一层：
          - VRAM不够？→ 近存储（FPGA/NDP）
          - 延迟太重要？→ FPGA（小batch无敌）
          - 构图太慢？→ GPU（45s vs 22min）
          - 功耗是红线？→ ASIC（5-6W vs 700W）
          ↓ "那未来呢？"
[终章]   所有方案的共同威胁是GPU的代际跃迁——但你可以在GPU之上叠加
          FPGA近存和ASIC加速器，形成一个混合架构
```

**贯穿全场的三个"元问题"**（每次提到一个方案，都用这三个问题拷问它）：

| 元问题 | 背后的洞察 |
|--------|-----------|
| **① 它比发布时的最新GPU快多少？** | 大多数论文用旧GPU基线，加速比在代际切换后快速贬值 |
| **② 这个优势3年后还成立吗？** | GPU H100→B200带宽涨2.4×，专用加速器的设计周期2-3年 |
| **③ 把它集成到现有系统的总成本是多少？** | 芯片级性能≠系统级TCO，软件/集成成本可能支配总支出 |

#### S2 · 问题尺度：从市场爆炸到规模爆炸

**叙事目的**：建立"紧迫感 + 规模分层认知"——不仅要让听众知道问题大，还要知道**不同量级需要不同的解决方式**。

**演讲结构**：

**第一步：市场曲线**（30 秒快速带过）
- 柱状图：$26.5亿→$89.5亿，CAGR 27.5%
- 一句话："市场在增长，这意味着你的客户数据也在增长"

**第二步：规模分层表（核心内容，2 分钟）**

```
            ┌────────────────────────────────────────────────────┐
            │  ≤10M     │  10M-100M  │  100M-1B   │  1B-10B+   │
────────────┼──────────┼───────────┼───────────┼───────────┤
内存需求    │  ~15 GB   │  ~150 GB   │  ~1.5 TB   │  ~15 TB    │
架构        │  CPU HNSW │  IVF-PQ    │  分布式    │  存算融合   │
GPU角色     │  不需要    │ 可选加速构图│  构图刚需   │ 仅构图     │
FPGA/ASIC   │  不需要    │  不需要     │ 开始有价值  │ 差异化方案  │
            └────────────────────────────────────────────────────┘
```

**第三步：一个"震撼"数据点**（让听众记住的锚）：
```
2022 年：大多数部署在 10M 级别 → GPU 是"可选项"
2024 年：主流部署向 100M-1B 迁移 → GPU 构图成为刚需
2026 年：头部企业触及 1B-10B → FPGA/ASIC 近存开始证明价值
```

**关键话术**：
```
向量数据库的架构不是"线性伸缩"的——
从 10M 到 100M，你遇到的是量化问题；
从 100M 到 1B，你遇到的是构图瓶颈；
从 1B 到 10B，你遇到的是数据搬运墙。

每个新量级都引入了"必须用新硬件才能解决"的问题。
硬件加速方案的出现时机不是巧合——它们是规模逼出来的。
```

> **过渡问题**："知道了规模效应——那为什么现有方案（尤其是 GPU）不能线性扩展去解决更大的规模？答案在下页。"

#### S3 · 打破假设：为什么堆算力不是答案

> **这是全场最重要的单页**。如果听众只记住一个概念，应该是这个。

**核心论点**：向量搜索的算术强度（FLOPs/byte）极低——**瓶颈在内存带宽，不在计算吞吐**。

**视觉**：一根柱状图

```
FLOPs/byte                         瓶颈类型
200+  ┃ ██ LLM Prefill             🟢 计算
 50   ┃ ██
 20   ┃ ██████████ LLM Decode      🟡 混合
 10   ┃ ████████████████ CNN       🟢 计算
  1   ┃
 0.75 ┃ ██ Vector Search (L2/IP)   🔴 内存带宽 ← 你在这里
  0   ┗━━━━━━━━━━━━━━━━━━━━━━━━
```

**讲稿脚本**：
- "Tensor Core 有数千 TFLOPS，但向量搜索用不上——因为它在等数据被搬过来"
- "这意味着：你花的 GPU 钱里，大部分买的是你不需要的东西（算力）"
- "所有高效的硬件方案——无论是 IBM 的 FPGA-SSD、Samsung 的 NAND 内计算、还是 DReX 的 CXL 加速器——都在做同一件事：**把计算往数据那里推，而不是把数据往计算这里拉**"

> **过渡**："知道了问题本质——数据搬运——现在来看看三个方案如何用不同的方式解决它。"

---

#### S4 · 八大瓶颈——但需要用"搬运"视角重新理解

**传统做法**：列 8 个痛点，逐个解释。

**更好的做法**：把 8 个痛点归类到 **"数据搬运"的统一框架**下：

```
搬运路径的每一段都有瓶颈：
                                   
[SSD/NAND] ──PCIe──→ [DRAM] ──PCIe──→ [GPU HBM] ──→ 计算
    ↑ ⑧数据搬运        ↑ ③内存墙      ↑ ①显存不够           ← 核心矛盾
                         ↑ ④资源竞争    ↑ ②RAG延迟反转
                         ↑ ⑤构图太慢    ↑ ⑦延迟抖动
                         ↑ ⑥TCO太高
```

**讲稿要点**：
- GPU 解决问题的思路是"把路修宽"（HBM 带宽翻倍）——但这受物理限制
- FPGA/ASIC 的思路是"把工厂搬到原料产地"——在 SSD 内部、在 DRAM 内部完成计算
- 从这个角度看，**三大路线的技术选择本质上是在不同搬运环节上做取舍**

#### S5 · 产业界策略光谱

**呈现**：一条横轴，从左到右标注"软件深度"到"硬件深度"

```
Pinecone ─ Qdrant ─ Milvus ─ Samsung ─ IBM
   ↓          ↓        ↓         ↓        ↓
  软件        混合     混合     存储内    全栈硬件
  优化        GPU     GPU+     计算      自研
                       FPGA    
```

**讲稿要点**：
- 每家厂商的选择不是"技术路线之争"，而是**对"数据搬运瓶颈在哪里"的不同判断**
- Pinecone 认为瓶颈在软件层（存算分离），IBM 认为瓶颈在物理层（近存储计算）
- **你的选择也应该基于你对自己瓶颈的判断**

> **第一幕结束过渡**："现在我们知道了：核心矛盾是数据搬运，不同厂商选择了不同位置来打破它。接下来我们逐一检视这三个选择——不是看谁更强，而是看每个选择的代价是什么。"

### 第二幕：三个答案——"每条路线的优势同时就是它的限制"（S6-S14）

这一部分的组织原则：**不是按"GPU→ASIC→FPGA"依次介绍产品，而是按"数据搬运距离"从长到短排序**——GPU 搬得最远（SSD→HBM），FPGA 搬得最短（SSD 内部完成），ASIC 介于之间。每个方案讲完后，用"三个元问题"审问。

| 页面 | 方案 | 搬运距离 | 押注的前提假设 |
|------|------|---------|-------------|
| S6 | 三路线总览（一张大图一张表） | — | — |
| S7-S8 | GPU（HBM 集中式计算） | 最长：SSD→DRAM→HBM | 假设 VRAM 够大、数据不需要频繁搬运 |
| S9-S10 | ASIC（存内/近存计算） | 中等：在存储器边界上计算 | 假设精度要求不高于 4-bit，或设计周期赶得上 GPU 迭代 |
| S11-S13 | FPGA（近存储计算） | 最短：在 SSD 内部完成 | 假设查询密度够高（10k+ 并发），或 batch 够小 |
| S14 | 加速定位总表 + 第一幕总回顾 | — | — |

#### S6 · 三路线总览（过渡页）

**叙事角色**：给出"阅读框架"，告诉听众接下来 8 页怎么听。

**呈现**：一张大表 + 每个方案一句话"本质"

| 路线 | 本质 | 一句话缺点（提前说） |
|------|------|-------------------|
| **GPU** | 把海量数据搬进 HBM，用成熟生态加速 | VRAM 是天花板，永远追不上数据增长 |
| **ASIC** | 把计算塞进存储器，消除搬运路径 | 功能固定，设计 3 年、流片即可能落后 |
| **FPGA** | 在 SSD 内部做全流水线搜索 | 大批量吞吐不如 GPU，需要 10k+ 并发才饱和 |

> **关键话术**："接下来每讲完一个方案，我都会问同一个问题——这个方案的优势在什么条件下会消失？不这样做，就会变成厂商的产品介绍。"

#### S7 · GPU：把数据搬到 HBM——"最快但最贵的数据搬运"

**叙事角色**：这是听众最熟悉的方案，所以**不讲它有多好，讲它的隐性代价**。

| 元问题 | 回答 |
|--------|------|
| ① 比发布时的最新 GPU 快多少？ | 构图 160× vs CPU——但这是 64 台 H100 vs 1 台服务器 |
| ② 3 年后还成立吗？ | VRAM 增长 2.4×/代，数据增长 2-3×/年——差距在拉大 |
| ③ 系统总成本？ | GPU 占自托管预算 80%+，软件生态成熟但硬件贵 |

**讲稿结构**（不按方案逐个讲，按 "三个角色" 讲）：

```
角色1：构图者（Index Builder）
  道具：CAGRA 45s vs HNSW 22min（10M）
  洞察：GPU 在这里无敌，但构图完 GPU 就闲置了
  → 引出的问题："花几十万买 GPU 只为每几天构图 45 秒？"

角色2：搜索者（Search Engine）
  道具：CAGRA 53× vs CPU（单查询）
  洞察：batch=100 时 49×，batch=1 时 3-9×
  → 引出的问题："你的工作负载 batch 是多大？"
  
角色3：混合者（Hybrid Architect）
  道具：VectorLiteRAG 的 20% 簇承载 93% 查询
  洞察：20% 的热数据住 GPU，80% 的冷数据住 CPU——**这是产业的无声共识**
  → 引出的问题："你没有意识到你并不需要全量 GPU 搜索，对吗？"
```

**关键数据打在大屏上**（每行一个，不要表格）：
```
CAGRA 构图 45s vs HNSW 22min       ✓ GPU 赢
单查询 CAGRA 3-9× vs CPU          ⚠️ 差距缩小
大批量 CAGRA 49× vs CPU           ✓ GPU 赢
混合方案 TCO 优于纯 GPU                 产业共识
```

#### S8 · GPU 路线——关键洞察提炼

**一页总结，不展示数据，展示判断**：

| # | 洞察 |
|---|------|
| 1 | GPU 在**构图**上的优势是其最不可替代的部分——因为它利用了 GPU 的大规模并行性做计算密集型工作 |
| 2 | GPU 在**在线搜索**上的优势被 VRAM 容量、资源竞争和 PCIe 搬运成本严重制约 |
| 3 | 业界"沉默的共识"是 GPU 构图 + CPU 查询——但厂商不公开说，因为不好营销 |
| 4 | Tensor Core 距离计算（QTT 的 38ms/1B/100% recall）可能改变规则——如果被验证和量产 |

> **过渡**："GPU 的思路是把数据搬过来算。那如果我们反过来——不让数据动，让计算住过去呢？"

#### S9 · ASIC：把计算塞进存储器——"最节能但最不灵活"

**叙事角色**：ASIC 在此领域最大的问题是"流片太慢，赶不上 GPU 迭代"，所以这一节要**讲"设计周期"和"精度税"这两个非技术听众不太关注但致命的问题**。

**先讲"这个方向为什么成立"**：

| 方案 | 在消除哪段搬运 | 节能多少 |
|------|-------------|---------|
| HERMES | DRAM↔ALU 的反复搬运 | 6.5W 跑 63.1 TOPS（vs H100 700W） |
| NDSEARCH | NAND↔主机的 PCIe 搬运 | 26.3W 整系统（vs GPU 700W） |
| ANNA | DRAM↔CPU 的 PQ 码本搬运 | 5.4W 完成 PQ 搜索 |

**再讲"这个方向的致命问题"**：

**问题一：精度税**
```
HERMES   ~3-4 bit       适合粗筛    需要 FP32 精修
Spyre    FP32 精确      适合兜底    延迟秒级
         你需要哪种？还是两者都要？
```

**问题二：设计周期 vs GPU 迭代**
```
ASIC 设计：需求→架构→RTL→验证→流片→测试 = 2-3 年
GPU 迭代：H100(2022)→B200(2024)→Rubin(2026) = 18 月一代
                    ↓
你在流片上花的每一块钱，都在赌 3 年后 GPU 没追上
```

---

#### S10 · ASIC 路线——关键洞察提炼

| # | 洞察 |
|---|------|
| 1 | ASIC 的能效比（HERMES 5.4W 63.1 TOPS）在任何一个单一指标上都是最优的，但代价是**精度受限和功能固定** |
| 2 | 该领域**6 个 ASIC 方案只有 2 个已流片**（HERMES、Spyre），其余停留在仿真——从论文到芯片的 gap 被严重低估 |
| 3 | ASIC 的核心机会不在云端通用搜索，而在**功耗严格约束的特定场景**（边缘、嵌入式、近存储） |
| 4 | DReX 的 PFU→NMA 两阶段设计（先低成本过滤、再精确计算）是通用价值最高的架构思想——可被其他路线借鉴 |

> **过渡**："ASIC 追求极致能效但牺牲了灵活性，FPGA 尝试在两者之间找到平衡——随需重构，也能近存计算。"

#### S11 · FPGA：把计算放进 SSD——"最短的搬运路径"

**叙事角色**：FPGA 的独特故事在于 **"双模"（FCM↔VSM）**——同一块硬件在存储和搜索之间切换。这不是技术选择，是经济学选择。

**故事结构**：

```
SSD 的日常：作为存储设备，大部分时间空闲（利用率 30-50%）
                ↓  
VSM 模式：SSD 控制器 FPGA 被重配置为搜索加速器
                ↓
            不需要买新硬件，不需要额外 PCIe 插槽
                ↓ "这不只是节能，这是省钱"
```

**VSM 全流水线图**（建议全页宽）：

```
主机 → PCIe（仅传输 查询向量 + 接收 K 个 ID）
  ↓
┌────── VSM (FCM SSD 内部 FPGA 流水线) ──────────────────┐
│  NAND Flash (32 并行通道, SLC 模式)                      │
│     → ECC Decode                                         │
│     → 32× 并行 MAC 流水线 (Distance Computation)         │
│     → 硬件 KNN 排序引擎 (Max-Heap, K无关延迟)            │
│     → 返回 Top-K（仅 K 个 ID，原始向量不跨 PCIe）        │
└──────────────────────────────────────────────────────────┘
```

**关键对比**（传递这个"为什么重要"）：

```
传统 GPU 搜索：     SSD → PCIe → DRAM → PCIe → HBM → GPU → 结果
                    每次数据搬运：延迟+功耗+PCIe 带宽竞争
                    
VSM FPGA 搜索：   NAND → FPGA 流水线 → 结果
                  数据不离开 SSD 控制器
```

---

#### S12 · FPGA 亮点：Falcon——算法比硬件更重要

**叙事角色**：Falcon 的故事是"算法-硬件协同设计"的案例——DST 算法是为 FPGA 定制的，但其核心思想可能超越 FPGA。

**结构**：

```
BFS：遍历→同步→遍历→同步（频繁片上-片外交换）
DST：遍历→遍历→遍历→延迟同步（减少 1.7-2.9× 同步次数）
                                    ↓
                         这是算法创新，不仅是硬件加速
```

**关键问题**（问听众）：

```
如果 DST 被移植到 GPU（cuVS 的遍历器中实现类似机制），
Falcon 的 FPGA 优势会消失吗？
    ↓
DST 的学术价值可能比 Falcon 硬件更长久
    ↓
但这不否定 Falcon——它证明了算法-硬件协同设计的方向
```

#### S13 · FPGA 路线——关键洞察提炼

| # | 洞察 |
|---|------|
| 1 | FPGA 在小 batch、低延迟、低功耗场景有不可替代的优势（Falcon sub-ms、VSM <10ms @ 1M）|
| 2 | **近存储 FPGA（VSM/SmartANNS）是最具差异化价值的方向**——将计算嵌入 SSD 内部，GPU 无法复制此架构 |
| 3 | VSM 的双模设计（FCM↔VSM）的战略价值在于**不增加硬件采购成本释放算力储备**——解决的是经济学问题而非技术问题 |
| 4 | FPGA 面临的根本问题：批量吞吐上限（受 FPGA 逻辑资源限制），且 GPU batch 优势随代际扩大 |

> **过渡**："现在三个方案都看完了。每个都有优势，每个都有代价。问题不是'哪个最好'，而是'你的瓶颈在哪一层'。"

#### S15 · 证据质量分级——先学会怀疑数据

**叙事角色**：在给出任何结论之前，先教会听众一个**批判性阅读任何技术报告的方法**。

**核心论点**：该领域最大的问题不是缺乏方案，而是「95 条声明只有 17 条通过对抗验证」。

**呈现**：四级证据金字塔

```
证据等级         代表方案                 常见陷阱
              ┌────────────────────┐
    A级       │ ❌ 尚无            │ 独立第三方复现
   已复现     └────────────────────┘
              ┌────────────────────┐
    B级       │ Falcon, VSM,       │ GPU 基线过时
   物理实测   │ HERMES, SmartANNS  │ 工况选择性报告
              └────────────────────┘
              ┌────────────────────┐
    C级       │ DReX, ANNA,        │ 仿真假设偏离实际
   仿真/综合  │ NDSEARCH, Cosmos   │ 热/功耗模型不完整
              └────────────────────┘
              ┌────────────────────┐
    D级       │ SK Hynix "3×"     │ 选择性指标
   厂商宣称   │ 市场预测数据       │ 缺失比较基准
              └────────────────────┘
```

**讲稿要点**：
- 当你在论文里看到 "X 倍 vs GPU"，第一反应不是"好厉害"，而是"**它的 GPU 基线是哪一年发布的？**"
- Falcon 26.9× vs V100（2017）≠ Falcon 26.9× vs H100（2022）——后者可能只有 3-5×
- 这不是学术不端，这是领域普遍存在的"基线选择偏差"

#### S16 · GPU 进化威胁模型——"加速比的半衰期"

**叙事角色**：这是整场报告最"insightful"的一页——揭示一个被几乎所有论文忽略的时间维度。

**一条线**：GPU 带宽演进曲线：

```
2017 V100:   ~900 GB/s     ┤
2020 A100:   ~2,000 GB/s   ┤ ████
2022 H100:   ~3,350 GB/s   ┤ ███████
2024 B200:   ~8,000 GB/s   ┤ ████████████████
2026 Rubin:  ~10,000+ GB/s ┤ ████████████████████
```

**三层含义**（一页讲三层，层层递进）：

| 层 | 含义 | 对决策的影响 |
|----|------|------------|
| **事实层** | GPU 带宽 9 年涨了 11× | 专用加速器论文的加速比以 GPU 发布年份计算 |
| **分析层** | Falcon 的 26.9× vs V100 → 到 B200 时可能只剩 3-5× | 所有声称的加速比都在随时间贬值 |
| **判断层** | ASIC 2-3 年设计周期，FPGA 1 年重配置，GPU 18 月一迭代 | FPGA 的可重构性在此场景是战略优势（可以跟进） |

**讲稿脚本**：
```
这是一个时间维度的分析，几乎所有论文都不讨论它——

当 ANNA 在 2022 年发表时，它对比的是 V100。
到 2024 B200 发布时——ANNA 的四次流片了吗？没有，它还在仿真。
到 2026 年你正在评估是否要采用这个方案时——GPU 又进了一代。

这意味着：所有专用加速器的"相对优势"都预设了 GPU 停在原地等它。
而 GPU 没有停。
```

#### S17 · 你的瓶颈在哪一层？——决策框架

**叙事角色**：取代传统的"选型决策树"，用一个**层级瓶颈分析框架**。

**呈现**：分层架构图，每层标注"如果瓶颈在这里→选择方向"

```
如果瓶颈在这里...        你的选择...
┌────────────────────────────────────────────────┐
│  ⑦ 需要 100% recall？                          │
│  → Spyre（ASIC 精确）或 DReX（CXL 精确NN）      │
├────────────────────────────────────────────────┤
│  ⑥ 功耗预算严格（<50W）？                       │
│  → HERMES（6.5W）或 ANNA（5.4W）或 VSM（25W）   │
├────────────────────────────────────────────────┤
│  ⑤ 单查询延迟要求 <5ms？                       │
│  → Falcon（FPGA, DST sub-ms）或 VSM（<10ms）    │
├────────────────────────────────────────────────┤
│  ④ 大批量吞吐优先？                            │
│  → CAGRA（GPU, batch=100 时 49× vs CPU）        │
├────────────────────────────────────────────────┤
│  ③ 构图太慢是瓶颈？                            │
│  → CAGRA（GPU, 45s vs 22min）或 CAS（H200）     │
├────────────────────────────────────────────────┤
│  ② 数据 > 单 GPU VRAM？                       │
│  → VSM（6B, 近存储）或 CAS（100B, 存储内搜索）  │
├────────────────────────────────────────────────┤
│  ① 还没遇到瓶颈，但想跟上趋势？                 │
│  → 关注 CXL 生态（Cosmos/DReX）和 QTT Tensor Core  │
└────────────────────────────────────────────────┘
```

**讲稿要点**：
- 这不是"选哪个方案"，而是"**你的瓶颈在哪一层，然后看谁解决那一层**"
- 多数企业现在的瓶颈在 **②③**（数据超过 VRAM，构图慢），所以 GPU 构图 + CPU 查询是合理起点
- 如果你的瓶颈在 **⑥**（功耗），你已经不需要考虑 GPU 了
- CXL 生态和 QTT 是值得关注的"未来信号"——现在不是选择但可能改变游戏规则

| 问题 | 建议回答方向 |

| "所以到底选 GPU 还是 FPGA？" | 没有普适答案。用决策树引导——关键是数据规模、延迟要求和功耗预算。强调**混合方案**（GPU 构图 + FPGA/CPU 查）正在成为趋势 |
| "这些数据可信吗？" | 坦诚标注各方案验证状态（实测/仿真/厂商数据）。强调**独立第三方复现普遍缺失**，所有对比有基线差异 |
| "国内有什么相关进展？" | 此次研究未覆盖国内厂商（如华为昇腾/百度昆仑/阿里含光），可作为独立课题 |

| "CXL 方案什么时候能用上？" | DReX / Cosmos 目前处于学术仿真阶段。CXL 3.0 生态正在成熟，预计 2027-2028 年可能出现原型产品 |
