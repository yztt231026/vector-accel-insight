#### S3 · 八大技术瓶颈——为什么 GPU 不够用

**展示方式**：8 张卡片 / 两行四列，每张含图标 + 一句话 + 关键数字

| # | 痛点 | 一句话 | 关键数字 |

#### S4 · 向量检索的计算特征——"为什么堆算力不是答案"

**核心论点**（这一页需要讲透，它是整个报告的技术前提）：

**建议呈现**：一张柱状图对比各工作负载的算术强度

```
FLOPs/byte
200+  │  ██ LLM Prefill
      │
 50   │  ██
      │
 20   │  ██████████ LLM Decode
 10   │  ████████████████ CNN Inference
      │
  1   │
 0.75 │  ██ Vector Search (L2/IP)    ← 🔴 纯内存受限
  0   └──────────────────────────
```

**讲稿要点**：
- 向量搜索的瓶颈在**内存带宽**，不在**计算吞吐**
- Tensor Core 数千 TFLOPS 对向量搜索**几乎无用**
- 这就是为什么几乎所有硬件方案（VSM / NDSEARCH / DReX / HERMES / Falcon）都围绕**缩短数据路径**设计
- → 自然引出下一部分："各方案如何缩短数据路径"

#### S7 · GPU 路线：IBM CAS —— 100B 向量的存储-计算融合

**核心观点**：GPU 不直接服务在线查询，而是加速**索引构建 + 嵌入生成**，搜索在存储层内完成。

**架构图建议**：

```
文件写入 → Storage Scale (GPFS)
    → Docling 解析 → Granite Embedding（6× H200 NVIDIA NIM）
    → 分层 ANN 索引（独立子索引，错误隔离）
    → 查询嵌入 → 存储层 ANN 搜索 → LLM 上下文注入
                    （不占用 GPU 显存）
```

**关键数字**：100B 向量 / 694ms 端到端延迟 / 120 天→4 天索引构建 / 未来目标 50-100ms

**弹框/标注**：还提供 Spyre ASIC 精确暴力搜索（10 亿/数秒/100% recall）

---

#### S8 · GPU 路线：CAGRA / Faiss —— 成熟生态的边界

**核心观点**：GPU 在**索引构建**上有绝对优势（45s vs 18-22min），但在**服务查询**上受制于 VRAM。

**核心对比表格**：

| 对比维度 | GPU 索引构建 | GPU 查询服务 | CPU 查询服务 |
|---------|------------|------------|------------|
| 吞吐（1536D-5M）| — | 3,200 QPS | 1,200 QPS |
| 月度成本（@100K QPS）| — | **$3,600**（6 GPU实例） | $28,800（48 CPU实例）|
| VRAM 容纳 | — | ~21M 768D FP16/GPU | 无限制 |
| 功耗 | 高（~700W） | 高 | 低 |

**讲稿要点**：GPU 成本降低 + 性能提升 ≠ 最优解，还要考虑与 LLM 推理的显存竞争（RAG 场景尤为关键）。

#### S9 · ASIC 路线总览：极致功耗 vs 生态碎片

| 方案 | 年份 | 工艺 | 功耗 | 核心加速 | 流片? |
|------|------|------|------|---------|-------|
| ANNA | 2022 | 40nm | **5.4W** | PQ 距离计算 | ❌ |
| IBM HERMES | 2023 | 14nm | **6.5W** | 模拟 MVM（63.1 TOPS） | ✅ |
| NDSEARCH | 2024 | 32nm | 26.3W | NAND 内 LUN 级计算 | ❌ |

| Cosmos | 2025 | — | — | CXL 并行距离 | ❌ |
| IBM Spyre | 2026 | — | ≈75W | 精确暴力搜索 + 嵌入 | ✅ |

**讲稿要点**：
- **HERMES** 和 **Spyre** 是唯二流片的 ASIC，但功能定位不同（MVM vs 精确搜索）
- 其余方案在仿真阶段，差距 vs GPU 可能随 GPU 迭代缩小
- ASIC 面临的根本挑战：**2-3 年设计周期 vs GPU 18 月一代的迭代速度**

---

#### S10 · ASIC 亮点方案逐讲

**DReX —— 最巧妙的架构设计**：

```
查询向量 → ① DRAM 内 PIM Filtering Units (PFU)
             符号位一致性过滤（快速淘汰非候选）
              → ② Near-Memory Accelerator (NMA)
                精确 L2/Cosine 距离计算
                 → Top-K 结果
```

- 核心设计思想：**用低成本计算先过滤掉大部分向量**，再对剩余小部分做精确计算
- 性能：24× vs CPU 方案，6.7× vs H100
- 架构：CXL Type-3，符合数据中心内存分解趋势

**IBM Spyre —— 精确搜索的最后一道防线**：
- 12 卡服务器数秒完成 10 亿向量 100% recall 精确搜索
- 在 IBM CAS 中作为 ANN 的补充（双轨策略）

---

#### S11 · FPGA 路线总览：重构性与近存储的天作之合

**核心观点**：FPGA 在 **小 batch / 低延迟 / 低功耗 / 近存储** 场景有独特优势。

**呈现方式**：对比表格 + 关键数据

| 方案 | FPGA | 核心加速 | 功耗 | 性能亮点 | 规模 |
|------|------|---------|------|---------|------|
| **IBM VSM** | FCM 内嵌 | **全流水线**（读取→ECC→距离→排序） | ≈25W | >17K QPS, 30× 能效 vs GPU | **6B** |
| **Falcon** | Alveo U250 | **DST 图遍历算法** | 55-62W | 4.3-19.5× 延迟↓, 26.9× 能效 | 10M |
| **FANNS** | Alveo U55c | IVF-PQ 全流程 | ~75W | 20.79× 吞吐 vs 基线 | 1B |
| **SmartANNS** | Kintex US+ | 分层索引筛选 | ~30W | 10.7× QPS, 80% 距离↓ | **1B** |

---

#### S12 · FPGA 亮点方案：IBM VSM —— 最完整的 FPGA 加速覆盖

**架构图**（建议全页宽）：

```
主机发送查询向量
    ↓ PCIe（仅传输查询 + 接收 K 个 ID）
VSM (FCM SSD 内部 FPGA 流水线)：
    ┌──────────────────────────────────────────┐
    │ NAND Flash (32并行通道, SLC模式)          │
    │    → ECC Decode                          │
    │    → 32× 并行 MAC 流水线（距离计算）       │
    │    → 硬件 KNN 排序引擎（Max-Heap）         │
    │    → 返回 Top-K（仅 K 个向量 ID）          │
    └──────────────────────────────────────────┘
    ↳ 数据**不经过 DRAM、不经过 PCIe** ← 关键
```

**三个核心数字**：
- **3500×** LPT 翻译表缩小 → 地址翻译不再是瓶颈
- **32** 条并行流水线 → K=1 到 K=100 **延迟相同**
- **48×** 近线性扩展 → 可部署为大规模集群

---

#### S13 · FPGA 亮点方案：Falcon —— 算法-硬件协同设计

**核心贡献**：DST（延迟同步图遍历）算法是专为 FPGA 有限片上资源设计的遍历策略。

**对比动画**（建议）：

```
BFS（传统）：每一步扩展后同步 → 频繁片上-片外数据交换
DST（Falcon）：延迟同步，允许"过时"向量继续遍历 → 减少同步次数

结果：1.7-2.9× 加速 + R@10 提升 0.14-4.93%
```

**关键局限**（建议用警示框标注）：
- ⚠️ GPU 基线为 V100（2017），对比 H100/B200 差距会缩小
- ⚠️ 10M 规模上限（FPGA 片上内存限制）
- ⚠️ 大 batch 下 GPU 反超（12× 内存带宽优势）

| # | 趋势 | 一句话判断 |

## 五、可能的 Q&A 预判

| 问题 | 建议回答方向 |
|------|------------|
| "所以到底选 GPU 还是 FPGA？" | 没有普适答案。用决策树引导——关键是数据规模、延迟要求和功耗预算。强调**混合方案**（GPU 构图 + FPGA/CPU 查）正在成为趋势 |
| "这些数据可信吗？" | 坦诚标注各方案验证状态（实测/仿真/厂商数据）。强调**独立第三方复现普遍缺失**，所有对比有基线差异 |
| "国内有什么相关进展？" | 此次研究未覆盖国内厂商（如华为昇腾/百度昆仑/阿里含光），可作为独立课题 |
| "这不就是存算一体吗？" | 部分对（NDSEARCH / HERMES 是存算一体），但三大路线的架构思路差异很大——FPGA 的可重构性、ASIC 的极致能效、GPU 的生态成熟度各不相同 |
| "CXL 方案什么时候能用上？" | DReX / Cosmos 目前处于学术仿真阶段。CXL 3.0 生态正在成熟，预计 2027-2028 年可能出现原型产品 |
| "IBM VSM 是产品还是研究？" | 目前是研究院项目/技术概念验证，尚未作为正式产品发布。其技术思路（FPGA 近存流水线）已被 ETH Zurich 的 Falcon/FANNS 在学术层面验证 |
